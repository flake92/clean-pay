# План сохранения текущего визуального дизайна

## Цель

Создать не художественный эскиз, а проверяемую копию текущего пользовательского интерфейса, достаточную для переноса дизайна в новую технологию без исходных frontend-файлов.

## Обязательный набор артефактов

Для каждого маршрута PAGE-001…PAGE-019 должны существовать:

1. карточка экрана с точной структурой и размерами;
2. перечень всех визуальных и интерактивных состояний;
3. эталонные изображения desktop и mobile с зафиксированными viewport, масштабом и состоянием данных;
4. запись использованных дизайн-токенов;
5. соответствующая сцена интерактивного макета;
6. отчёт визуального сравнения и перечень отличий;
7. трассировка каждого элемента управления до логической и HTTP-операции.

## Метод проверки

- запускать исходный интерфейс с контролируемыми mock-данными;
- снимать одинаковые состояния при одинаковом viewport и device scale factor;
- снимать макет при тех же условиях;
- выполнять наложение или pixel-diff;
- вручную проверять адаптивные переломы, переполнение текста, клавиатурный фокус, loading и ошибки;
- повторять после исправления до отсутствия необъяснённых расхождений.

## Запрет ложной готовности

Текущий `mockup/index.html` считается черновиком до выполнения этого плана. Его наличие не доказывает сохранение дизайна. В финальном манифесте должны быть перечислены реальные эталоны и результаты сравнения.

## Фактически выполнено

- записаны 38 эталонов: PAGE-001…PAGE-019 в desktop и mobile;
- зафиксированы viewport, сцены, guard/redirect и SHA-256 каждого файла;
- фактически пройдена цепочка регистрации с доставкой SMTP-письма, подтверждением кода, bootstrap-сессией, Passkey-экраном, кабинетом и выходом;
- отдельно снят SSR-экран Telegram WebApp до получения внешнего Telegram-контекста; защитный переход без контекста подтверждён;
- подтверждены две оболочки, основные responsive-переломы и визуальные токены.

## Текущий статус

`ЭТАЛОНЫ ТЕКУЩЕГО UI ЗАФИКСИРОВАНЫ; PIXEL-DIFF НОВОЙ РЕАЛИЗАЦИИ НЕВОЗМОЖЕН ДО ПОЯВЛЕНИЯ НОВОЙ РЕАЛИЗАЦИИ`.

Это не блокирует удаление старого frontend после закрытия остальных ворот: эталоны, экранные контракты и mock-сцены сохраняются независимо. Финальный visual gate новой реализации закрывается позднее сравнением с данным набором.
