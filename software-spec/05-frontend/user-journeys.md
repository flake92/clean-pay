# Пользовательские трассы

## Покупка через e-mail

Вход → ввод e-mail → определение существования → форма пароля или регистрации → создание cookies/пользователя. Неподтверждённый адрес ведёт на экран кода: запрос/повторная отправка → ввод шести цифр → подтверждённый доступ. Далее: «Тарифы» → точное предложение duration+gateway → `/payment` → повторное получение предложений → покупка со стабильным ключом → провайдер/возврат → опрос статуса → кабинет. Любое изменение предложения останавливает оплату и просит повторно подтвердить выбор.

## Telegram и объединение

«Войти через Telegram» → начало OIDC → официальный провайдер → callback → локальная и внешняя сверка. При столкновении владельцев открывается `/link-account`: пользователь видит обе стороны, подтверждает объединение, система выполняет owner-fenced перенос и возвращает в кабинет; отмена очищает доказательство. Telegram WebApp автоматически отправляет подписанный `initData` и использует только проверенный безопасный redirect.

## Управление подпиской

Кабинет загружает live URL, устройства, предложения, историю и поддержку. «Подключить»/«Копировать» меняют только состояние браузера; перевыпуск изменяет upstream; удаление устройства изменяет upstream и затем обновляет список; промокод показывает награду/ошибку; продление выбирает предложение, создаёт долговечную операцию и проходит gateway/status.

## Аккаунт

Профиль → повторная отправка/смена e-mail → подтверждение; смена пароля отзывает соседние сеансы. «Связать аккаунт» предлагает e-mail credentials, Telegram linking/recheck и настройку/list/delete ключей доступа. Каждая мутация блокирует свою кнопку до результата и показывает inline success/error, не стирая введённые данные без необходимости.

## PWA

Страница установки вызывает native prompt, когда он доступен; для iOS, embedded browser и Android fallback показывает соответствующую инструкцию. В установленном режиме основное действие открывает кабинет. Service worker кэширует только публичную оболочку; offline-навигация ведёт на `/offline`; обновление активирует свежий worker без кэширования приватных ответов.

Каждая трасса проверяется как `экран → control → действие → логическая операция → HTTP-карточка → изменение состояния → результат → следующий экран`. Детали связаны через `api-usage.md` и `pages/PAGE-*.md`.
