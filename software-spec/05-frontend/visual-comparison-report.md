# Отчёт визуальной сверки

## Проверенные наборы

- эталон действующего приложения: `reference/current/desktop` и `reference/current/mobile`, 19 + 19 JPEG;
- автономный макет: `mockup/index.html`;
- контрольный рендер макета: `reference/mockup/desktop` и `reference/mockup/mobile`, 19 + 19 JPEG;
- размеры окна: desktop `1440×1000`, mobile `390×844`, масштаб 100 %;
- дата повторного рендера: 22 июля 2026 года.

Оба набора имеют `SHA256SUMS`; проверка всех 76 файлов прошла. Макет открывался из локального `file://`, то есть для его структуры и интеракций не требуется приложение или сервер.

## Автоматическая проверка каждого экрана

Для всех 38 рендеров макета подтверждено:

- маршрут открывается и имеет ожидаемый H1;
- JavaScript не выдаёт page error;
- отсутствует горизонтальное переполнение в заданном viewport;
- служебная панель состояний и selector маршрутов скрыты в обычном режиме;
- AuthShell/AppShell, desktop/mobile breakpoint, шрифт Inter, логотип, палитра, карточки и формы загружаются автономно.

Служебные controls доступны только с query `?controls=1`; поэтому они не попадают в обычный визуальный контракт.

## Исправленные расхождения макета

В ходе сверки устранены: sidebar 260 → 300 px; topbar 70 → 80 px; неверные offsets оболочки; дублированный hero главной; фиктивная активная Premium-подписка; фиктивные планы/готовая оплата; неверные сцены подтверждённого e-mail, профиля, способов входа, поддержки и возврата платежа; видимая служебная toolbar; отсутствие Inter; горизонтальное переполнение кабинета на mobile.

Текущие демонстрационные сцены теперь соответствуют сценам эталонного манифеста: пустые тарифы, отсутствующий выбор оплаты, отсутствие подписки, проверка платежа без локальной ссылки, подтверждённый e-mail, неподключённые Telegram/Passkey и опубликованные кнопки поддержки.

## Числовая санитарная метрика

После приведения контрольного рендера к размеру соответствующего эталона среднее сходство по абсолютной разности RGB составляет 95,86 % по 38 парам; минимум 89,49 % у mobile PAGE-007. Эта величина приводится только как воспроизводимая санитарная проверка общего фона/геометрии. Она не является perceptual pixel-diff и не доказывает точное совпадение иконок, переносов или каждой строки.

## Авторитет и допустимые различия

При конфликте авторитетны в следующем порядке:

1. JPEG действующего приложения в `reference/current`;
2. точная структура, состояния и строки `pages/PAGE-*.md`;
3. токены `design-tokens.md`;
4. автономный макет и его контрольные JPEG.

В макете системные/PrimeIcons частично заменены простыми контурными знаками, а неиспользуемые в эталонной сцене ветви выбираются служебным режимом. Поэтому финальный визуальный барьер будущего Ruby-интерфейса должен сравнивать его непосредственно с `reference/current`, а не принимать сам макет за пиксельный oracle.

## Итог текущего этапа

Визуальный дизайн физически сохранён: точный raster-эталон каждого маршрута в двух viewport, карточка поведения каждого экрана, токены, автономный кликабельный макет и повторный браузерный рендер. Финальный барьер новой реализации остаётся будущей проверкой: Ruby-приложение ещё не существует, поэтому его pixel-diff выполнить сейчас невозможно.
