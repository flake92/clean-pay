# Модель хранения

## Постоянное хранилище

Основное транзакционное хранилище — PostgreSQL. Оно обязано поддерживать транзакции, блокировку строк, уникальные ограничения, внешние ключи, JSON, массивы строк, двоичные открытые ключи, десятичные суммы и временные метки с миллисекундной точностью. Строка подключения задаётся `DATABASE_URL`.

Redis — ускоряющее, но не авторитетное хранилище: счётчики ограничения частоты и общий кэш readiness. Недоступность Redis не должна уничтожать доказательства антизлоупотребления: долговечные события сохраняются в PostgreSQL и служат резервным механизмом/аудитом.

## Локальные и внешние данные

Локально материализуются пользователи, сеансы, одноразовые церемонии, платёжные команды и история платежей. Тарифы, текущая подписка и устройства читаются из Remnashop/Remnawave и не имеют локальных таблиц. `IntegrationStatus` — только последний эксплуатационный снимок, а не авторитетная проверка доступности.

## Логическое и физическое владение

Зашифрованные токены Remnashop физически лежат в строке одного сеанса, но логически принадлежат пользователю. Поэтому любая реализация обязана сохранять атомарную передачу токенов между сеансами и очистку дубликатов. Платёжная запись физически хранит снимок ответа, но авторитетным состоянием оплаты остаётся провайдер; синхронизация может обновить снимок.

## Время и JSON

Все сроки и моменты сохраняются в UTC. JSON-поля используются только для версионируемых снимков запроса/ответа/ошибки и очищенных метаданных; критические ключи поиска и состояния обязаны оставаться типизированными столбцами. Изменение структуры JSON должно быть обратно совместимо с уже сохранёнными строками.
