# Ограничения данных

## Ограничения базы

- Первичные ключи обязательны для всех сущностей; `PaymentHistorySyncState.userId`, `AppSetting.key` — естественные первичные ключи.
- Уникальны внешние идентичности пользователя, отпечатки одноразовых секретов, credential ID, внешний ID платежа, внешний ключ операции, связь записи с операцией и имя статуса интеграции.
- Уникальна пара `(userId, idempotencyKeyHash)`.
- Пары токена захвата и срока аренды в операции/синхронизации одновременно заполнены либо одновременно пусты.
- Счётчики попыток, ошибок и поколение неотрицательны.
- Денежная сумма платёжной записи имеет точность `DECIMAL(12,2)`.

## Ограничения приложения

- E-mail нормализуется до поиска и сохранения владельца.
- Все одноразовые значения проверяются по отпечатку, сроку, использованию и, где применимо, числу попыток.
- Ограждение владельца запрещает перенос или платёж при изменившейся привязке.
- Идемпотентный ключ с другим отпечатком запроса возвращает конфликт.
- Нельзя удалить последний ключ доступа.
- Входная подтверждённая сумма допускает до 8 дробных знаков для точного сравнения с предложением; сохраняемый окончательный результат обязан помещаться в `DECIMAL(12,2)` без неявного округления.
- `duration_days` синтаксически допускает 0..365000, но затем обязан точно входить в выбранное актуальное предложение.
- Неизвестные поля JSON-запроса не считаются автоматически запрещёнными, если конкретная HTTP-карточка не требует обратного.
