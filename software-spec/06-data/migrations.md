## Требования к новой реализации

Новая реализация может создать итоговую схему напрямую только для совершенно новой базы. Для существующей базы она обязана либо воспроизвести эквивалентные переходы, либо предоставить отдельно проверенный план переноса, сохраняющий каждое поле, ограничение и строку. Версия схемы должна записываться так, чтобы повторный запуск не применял шаг дважды.

## Безопасность миграций

- Перед `NOT NULL` сначала добавить nullable-поле, заполнить старые строки, проверить результат и только затем ужесточить ограничение.
- Смена типа Telegram ID обязана завершаться отказом, а не потерей malformed-значений.
- Удаление индекса не должно ждать блокировку бесконечно.
- DDL выполняется транзакционно там, где это поддерживает база.
- Резервная копия и проверка восстановления обязательны перед миграцией production с изменением типа, удалением столбца или массовым backfill.
- Rollback приложения допускается только если старая версия совместима с новой схемой; обратная DDL-миграция с потерей данных запрещена.
