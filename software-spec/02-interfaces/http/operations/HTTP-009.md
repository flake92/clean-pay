# Начать смену e-mail

## Идентификатор

`HTTP-009`

## Назначение

Установить новый pending e-mail во внешней системе, снять локальный признак подтверждения и отправить код на новый адрес.

## Владелец

Модуль доступа и идентичности.

## Акторы

Полностью вошедший пользователь.

## Предусловия

Полная сессия с разрешённой внешней учётной записью; при включённом Turnstile — token.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `email` | string | да | нет | e-mail | внешняя схема trim/lower-case, ≤255 | нет | новый pending адрес |
| два поля Turnstile | string | нет | нет | token | удаляются перед внешним вызовом | нет | антибот |

Иные дополнительные поля передаются внешней схеме и игнорируются.

## Текущий транспорт

`POST /api/bff/auth/email/change`; JSON UTF-8 до 65 536 байт; доверенный origin; cookie сессии.

## Правила валидации

Turnstile → полная внешняя авторизация → cooldown 1/60 секунд по user ID → общий лимит 5/900 секунд по новому e-mail и Telegram ID → внешняя схема.

## Нормализация

Rate identity trim/lower-case; e-mail wire normalization выполняет Remnashop.

## Авторизация

Proxy требует обычную сессию; сервер не задаёт `allowUnverifiedEmail`, поэтому действует полная политика e-mail/Telegram и согласованность внешнего владельца.

## Идемпотентность

Нет. Повтор блокируется cooldown; внешняя система может вернуть конфликт или заменить pending адрес.

## Основной сценарий

1. Проверить запрос, Turnstile, сессию и лимиты.
2. Вызвать RS-011.
3. Установить локальный `emailVerified=false` и обновить access-cookie.
4. Вызвать RS-009 для возвращённого `pending_email`, с разрешёнными 3 попытками временной ошибки SMTP.
5. Записать audit и вернуть оба результата.

## Альтернативные сценарии

Нет успешной ветки без запроса письма.

## Ошибочные сценарии

Стандартные `400`, `401`, `403`, `409`, `413`, `415`, `429`, `502` Remnashop/SMTP, `503` Turnstile, `500` локального хранилища.

Критический частичный успех: после успешного RS-011 локальный признак подтверждения снимается **до** отправки письма. Если письмо не отправлено или audit падает, операция отвечает ошибкой, но новый pending e-mail уже установлен, пользователь может быть ограничен страницей подтверждения, а access-cookie уже обновлена. Компенсации к прежнему адресу нет.

## Логический результат

`200`:

```json
{
  "data": {
    "success": true,
    "pending_email": "new@example.com",
    "emailVerification": {
      "success": true,
      "target_email": "new@example.com",
      "expires_at": "2026-07-22T12:15:00Z"
    }
  }
}
```

Access-cookie переиздаётся с `ev:false`; refresh-cookie меняется только при общей rotation.

## Побочные эффекты

Новый внешний pending e-mail, локальное снятие подтверждения, письмо, rate-limit, cookie, audit.

## Транзакционные требования

Внешняя смена адреса, локальное изменение и письмо последовательны и не атомарны; компенсации нет.

## Наблюдаемость

Журналируются pending и target e-mail; token и код не журналируются.

## Источники

Доказательства находятся в `09-traceability/`.

## Статус уверенности

`подтверждено`
