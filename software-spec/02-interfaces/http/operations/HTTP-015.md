# Удалить один ключ доступа

## Идентификатор

`HTTP-015`

## Назначение

Удалить принадлежащий пользователю ключ, не позволяя удалить последний.

## Владелец

Модуль доступа и идентичности.

## Акторы

Полностью вошедший пользователь.

## Предусловия

FULL-сессия, политика e-mail/Telegram и не менее двух ключей владельца.

## Логический входной контракт

Path-поле `id:string`: внутренний локальный ID записи ключа, ровно один непустой URL-сегмент. Это не `credentialId` WebAuthn. Query и body не читаются.

## Текущий транспорт

`DELETE /account/passkeys/{id}`. Bodyless Rails resource mutation protected by CSRF; credential is owner-scoped.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Сегмент передаётся route decoder как строка. Явных length/charset ограничений нет; поиск точный.

## Нормализация

Нет.

## Авторизация

FULL-сессия; credential обязан принадлежать текущему user. Чужой ID не раскрывается и трактуется как 404.

## Идемпотентность

Нет в строгом HTTP-смысле: первый успех 200, повтор 404.

## Основной сценарий

1. В транзакции заблокировать строку пользователя `FOR UPDATE`.
2. Найти принадлежащий credential.
3. Посчитать все ключи владельца.
4. При count>1 удалить выбранный.
5. После commit записать audit `passkey_deleted` с wire credential ID.

## Альтернативные сценарии

Параллельные удаления сериализуются row lock: из двух последних ключей удалится только один, второй увидит count 1 и получит 403.

## Ошибочные сценарии

`401` нет session/user, `403 FORBIDDEN` origin/последний ключ/email policy, `404 NOT_FOUND` отсутствующий или чужой ID, `500` DB/audit. Транзакция ждёт максимум 5 секунд и имеет timeout 15 секунд; её отказ нормализуется в 500.

## Логический результат

`303 See Other` to `/link-account` with flash after deletion; last-key failure is rendered by Rails.
## Побочные эффекты

Удаление credential и audit. Если audit падает после commit, HTTP может быть 500, хотя ключ уже удалён; повтор вернёт 404.

## Транзакционные требования

Проверка количества и удаление атомарны под row lock пользователя.

## Наблюдаемость

Audit содержит credential ID, не public key.

## Источники

Доказательства находятся в `09-traceability/`.

## Статус уверенности

`требует повторной проверки после ADR-003`
