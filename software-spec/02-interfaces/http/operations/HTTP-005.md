# Завершить все активные локальные сессии пользователя

## Идентификатор

`HTTP-005`

## Назначение

Выйти из Clean Pay, отозвав все активные локальные сессии определённого пользователя и удалив cookie текущего браузера.

## Владелец

Модуль доступа и идентичности.

## Акторы

Пользователь или клиентский интерфейс, включая уже истёкшую/повреждённую сессию.

## Предусловия

Нет требования действующей сессии, но как browser mutation запрос требует доверенный источник.

## Логический входной контракт

Тело отсутствует и не читается. Полезными входами являются `Origin`/fallback `Referer`, `clean_pay_access` и `clean_pay_refresh`.

## Текущий транспорт

`POST /api/bff/auth/logout`. Это явно bodyless-операция: `Content-Type` не требуется. Query/path-параметры игнорируются. Источник обязан совпасть с публичным origin.

## Правила валидации

Если `Origin` присутствует, он имеет приоритет даже при невалидном значении; fallback к `Referer` происходит только при полном отсутствии `Origin`. Невалидный источник → 403.

## Нормализация

Нет.

## Авторизация

Путь публичный, чтобы позволить очистить повреждённые cookie. Валидная access-сессия определяет пользователя; иначе refresh token может определить одну сессию.

## Идемпотентность

Да по результату: повтор возвращает успех и повторно удаляет cookie.

## Основной сценарий

1. Создать audit action `auth_logout` до очистки сессии.
2. Если access token валиден и его активная запись найдена, установить `revokedAt` у **всех** активных сессий этого пользователя.
3. Если access не определил активную сессию, но есть refresh, найти соответствующую текущую либо ещё находящуюся в 10-секундном grace запись и отозвать найденную сессию.
4. Удалить обе cookie текущего браузера.
5. Вернуть успех.

## Альтернативные сценарии

Без cookie или с неизвестными cookie выполняется только их удаление. Ветка по refresh отзывает одну определённую сессию, тогда как валидный access отзывает все сессии пользователя.

## Ошибочные сценарии

| HTTP | Код | Условие |
|---:|---|---|
| 403 | `FORBIDDEN` | недоверенный источник; обработчик не запущен, cookie proxy не удаляет |
| 500 | `INTERNAL_ERROR` | аудит или хранилище завершились ошибкой |

Если аудит не записался, очистка не начинается. Если отзыв БД упал, удаление cookie в этой функции не гарантируется, потому что оно выполняется после БД-операций.

## Логический результат

`200 {"data":{"success":true}}`. Успешный ответ удаляет `clean_pay_access` и `clean_pay_refresh` посредством Set-Cookie с истёкшим сроком.

## Побочные эффекты

Audit record, отзыв одной или всех активных сессий, удаление cookie. Внешняя сессия Remnashop отдельно не отзывается по сети; её зашифрованные токены остаются в отозванных локальных записях до retention, если иная операция их не очистила.

## Транзакционные требования

Аудит, отзыв и удаление cookie не объединены одной транзакцией. Массовый отзыв выполняется одной update-many операцией.

## Наблюдаемость

Фиксируются наличие cookie, валидность access и способ отзыва (`access`, `refresh`, `cookies_only`) без содержимого token.

## Источники

Доказательства перечислены в `09-traceability/`.

## Статус уверенности

`подтверждено`
