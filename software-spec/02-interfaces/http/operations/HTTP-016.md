# Войти из Telegram WebApp

## Идентификатор

`HTTP-016`

## Назначение

Передать подписанные данные Telegram WebApp в Remnashop, доказать Telegram-профиль, согласовать локального владельца и создать веб-сессию.

## Владелец

Модуль доступа и идентичности.

## Акторы

Telegram WebApp, браузер внутри Telegram и пользователь.

## Предусловия

Непустое `window.Telegram.WebApp.initData`, сформированное Telegram и принимаемое Remnashop.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `initData` | string | да | нет | непрозрачная query-string Telegram | после trim непустая | нет | доказательство WebApp |
| `redirectTo` | string | нет | нет | безопасный root-relative path | начинается `/`, но не `//`; origin/absolute URL запрещены | `/cabinet` | продолжение после входа |

Дополнительные поля игнорируются. Wire к Remnashop переименовывает `initData` в `init_data`.

## Текущий транспорт

`POST /account/telegram_session`. Rails form scope `telegram_session[init_data,redirect_to]`; CSRF; `redirect_to` uses SafeReturnPath.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

После внешнего RS-004 Clean Pay вызывает RS-007; `telegram_id` обязан быть не `null`. Затем лимит 20/900 секунд по проверенному Telegram ID.

## Нормализация

Только trim `initData`; `redirectTo` проходит безопасную path policy.

## Авторизация

Публично; авторизация основана на проверке initData Remnashop и повторном чтении профиля.

## Идемпотентность

Нет: повтор создаёт новую локальную сессию.

## Основной сценарий

RS-004 → RS-007 → rate-limit → согласование локального user → создание FULL-сессии method TELEGRAM → при флаге recovery восстановление внешней Telegram session → ответ.

## Альтернативные сценарии

Согласование может объединить локальные идентичности либо пометить необходимость восстановления внешних токенов согласно правилам единственного владельца.

## Ошибочные сценарии

`400 BAD_REQUEST` с фиксированным сообщением при пустом initData; `401 UNAUTHORIZED` при профиле без Telegram ID; также `403` origin `429`, `502` Remnashop, `500`. `503/504` не формируются.

## Логический результат

`303 See Other` to the validated Rails page with newly issued session cookies.
## Побочные эффекты

Внешняя аутентификация, локальный user/session, внешние зашифрованные tokens, rate-limit, cookie и технический audit.

## Транзакционные требования

Внешняя проверка и локальная сессия не атомарны. Ответ с cookie создаётся до необязательного recovery; ошибка recovery может завершить запрос ошибкой после уже созданной записи сессии, но response object ещё не отправлен.

## Наблюдаемость

Фиксируется Telegram ID и user ID, но не initData/tokens.

## Источники

Доказательства находятся в `09-traceability/`.

## Статус уверенности

`требует повторной проверки после ADR-003`
