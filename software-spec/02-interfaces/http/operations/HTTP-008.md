# Подтвердить e-mail одноразовым кодом

## Идентификатор

`HTTP-008`

## Назначение

Подтвердить владение адресом, немедленно зафиксировать необратимый результат локально и затем попытаться безопасно согласовать e-mail/Telegram-владельца.

## Владелец

Модуль доступа и идентичности.

## Акторы

Пользователь, получивший письмо с кодом.

## Предусловия

Локальная сессия, внешняя связка token и, кроме already-verified ветки, действующий шестизначный код.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `code` | string | да | нет | ровно 6 ASCII-цифр | внешняя схема | нет | одноразовое доказательство |
| `email` | string | нет | нет | e-mail | внешний валидатор; Clean Pay может подставить вычисленный адрес | pending → current внешний → локальный | ожидаемый адрес |
| `registrationFlow` | boolean | нет | нет | boolean | удаляется и наружу не передаётся | отсутствует | только клиентский контекст |
| два поля Turnstile | string | нет | нет | token | удаляются | нет | антибот |

Иные дополнительные поля после удаления `registrationFlow` и Turnstile передаются Remnashop и игнорируются закреплённой схемой.

## Текущий транспорт

`POST /api/bff/auth/email/confirm`; доверенный origin; JSON UTF-8 до 65 536 байт; cookie сессии.

## Правила валидации

Turnstile выполняется до сессии. Target выбирается: явный `email`, затем `pending_email`, внешний current e-mail, локальный e-mail, иначе отсутствие. Лимит — 5 подтверждений на 900 секунд по target/Telegram ID.

## Нормализация

При проверке already-verified адрес сравнивается без учёта регистра. Внешняя схема нормализует e-mail. Код не trim’ится Clean Pay.

## Авторизация

Разрешена неподтверждённая e-mail-сессия. Внешние токены получаются с `allowUnverifiedEmail=true`.

## Идемпотентность

Условная. Если внешний профиль уже подтверждён и его адрес совпадает с target без учёта регистра, внешний confirm не вызывается и возвращается успешный `already_verified:true`. Уже потреблённый код при незавершённой внешней проверке повторно использовать нельзя.

## Основной сценарий

1. Проверить Turnstile, сессию, внешний профиль и лимит.
2. Если не already-verified, вызвать RS-010.
3. Немедленно после внешнего успеха определить внешний owner ID.
4. В одной локальной транзакции пометить существующего владельца e-mail подтверждённым, если он другой; текущему владельцу присвоить e-mail только если конфликтующего владельца нет; в любом случае установить `authPending` и pending evidence.
5. Обновить access-cookie и записать `email_verified`, если локальное состояние изменилось.
6. Под fence изменения владельца попытаться связать Telegram, при необходимости объединить внешних и локальных владельцев, присоединить токены и очистить pending.

## Альтернативные сценарии

- Уже подтверждённый совпадающий e-mail даёт синтетический успех без использования кода.
- Конфликт Telegram может вызвать admin merge Remnashop. Если у обеих внешних записей подписки, изменение владельца не выполняется.
- Любая ошибка шага 6 поглощается: подтверждение не откатывается, `account_sync_pending=true`, `authPending` остаётся.

## Ошибочные сценарии

До необратимого подтверждения возможны: `400 VALIDATION_ERROR`, `400 EMAIL_CODE_INVALID`, `400 EMAIL_CODE_EXPIRED`, `401`, `403`, `409` конфликта, `413`, `415`, `429`, `502` Remnashop, `503` Turnstile, `500` локального хранилища. После успешного внешнего confirm ошибки синхронизации **не** превращают HTTP-ответ в ошибку; они отражаются полем pending.

## Логический результат

`200`:

```json
{
  "data": {
    "success": true,
    "email": "user@example.com",
    "already_verified": false,
    "account_sync_pending": false
  }
}
```

Clean Pay всегда добавляет boolean `already_verified` и `account_sync_pending`, даже если внешняя схема их не возвращала. Возможна refresh/access cookie rotation.

## Побочные эффекты

Потребление внешнего кода; подтверждение/перенос pending e-mail; локальные evidence и признаки; возможное внешнее/локальное объединение владельцев; токены; audit; cookie.

## Транзакционные требования

Необратимый внешний confirm предшествует локальной транзакции evidence. После неё используется отдельная fenced операция согласования. Ошибка второй фазы не может отменять первую и не должна предлагать пользователю повторно вводить код.

## Наблюдаемость

Логи отличают внешний успех, локальную фиксацию и post-confirm sync failure. Код и токены не записываются.

## Источники

Доказательства находятся в `09-traceability/`.
