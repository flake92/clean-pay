# Получить историю платежей

## Идентификатор

`HTTP-031`

## Назначение

Синхронизировать и вернуть последние платежи только текущего доказанного upstream-владельца.

## Владелец

Модуль платежей.

## Акторы

Вошедший пользователь с подтверждённым e-mail.

## Предусловия

Полная сессия, Remnashop-связь и непротиворечивый payment upstream owner.

## Логический входной контракт

Входных данных нет; query и тело не используются.

## Текущий транспорт

`GET /api/bff/payments/history`; session cookie.

## Правила валидации

Upstream user ID извлекается из access token и сверяется с зафиксированным owner. Capability mode использует размер страницы `min(100,max_page_size)`; legacy mode получает совместимый список. Каждая транзакция проверяет UUID/владельца, временные метки и типы до записи.

## Нормализация

Статусы `PENDING|COMPLETED|FAILED|CANCELED|REFUNDED` выдаются в нижнем регистре, неизвестный — `unknown`; decimal становится строкой; даты — ISO-8601 UTC.

## Авторизация

Сессия, правило подтверждения e-mail, access token и owner fence.

## Идемпотентность

Повторное чтение выполняет upsert по `payment_id`, не меняет владельца и не откатывает более свежую upstream-версию.

## Основной сценарий

Определить capabilities; синхронизировать одну страницу либо legacy transactions; затем прочитать 20 записей пользователя по `upstreamCreatedAt DESC,paymentId DESC`.

## Альтернативные сценарии

Пустая история — `[]`. Capability endpoint может отсутствовать: это штатный переход на legacy API.

## Ошибочные сценарии

`401`, `403`, `409 CONFLICT` владельца/платежа, `502` upstream/contract, `500` БД. Ошибка синхронизации не возвращает устаревший список: весь запрос завершается ошибкой.

## Логический результат

`200 {"data":[{"id","payment_id","purchase_type","status","final_amount","currency","gateway_type","plan_code","plan_name","duration_days","device_limit","traffic_limit","is_free","created_at","updated_at"}]}`; nullable поля определены моделью платежа.

## Побочные эффекты

Обновление локальной истории/курсора и возможный refresh сессии.

## Транзакционные требования

Страница применяется атомарно под upstream-owner lock; конфликт вставки повторяется ограниченно, чужое владение никогда не переписывается.

## Наблюдаемость

Счётчики/режим sync и ошибки владельца без токенов и payment URL.

## Источники

Доказательства находятся в `09-traceability/`; внешние payment-capabilities/transactions и модель 03.

## Статус уверенности

`подтверждено`
