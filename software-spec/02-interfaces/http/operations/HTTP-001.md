# Определить наличие локальной учётной записи

## Идентификатор

`HTTP-001`

## Назначение

До показа формы входа определить, существует ли локальный пользователь с данным e-mail и зарегистрирован ли у него хотя бы один ключ доступа. Операция не проверяет пароль и не обращается во внешнюю систему.

## Владелец

Логический модуль доступа и идентичности.

## Акторы

Гость и браузерный интерфейс входа.

## Предусловия

Нет пользовательской сессии. Запрос должен пройти общую проверку источника и типа содержимого.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `email` | string | да | нет | произвольная строка, затем trim и lower-case | после нормализации непустая; синтаксис e-mail локально не проверяется | нет | ключ поиска и ограничения частоты |

Любые дополнительные form-поля допускаются и игнорируются. Отсутствие, `null`, boolean, number, object, array и строка из одних пробелов для `email` дают одинаковую ошибку валидации.

## Текущий транспорт

`POST /account/identity`. Rails form scope `identity[email]`; CSRF; unknown form fields are removed by strong parameters.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Пустое тело, невалидный JSON и невалидная UTF-8-последовательность → `400 VALIDATION_ERROR`. JSON `null`, массив или примитив → `400 VALIDATION_ERROR`. Слишком большое тело → `413 VALIDATION_ERROR`. Неверный media type → `415 VALIDATION_ERROR`. Недоверенный источник → `403 FORBIDDEN` до разбора тела.

## Нормализация

`email = StringValue.trim().toLowerCase()`. Unicode и внутренние пробелы дополнительно не преобразуются.

## Авторизация

Публичная операция. Наличие cookie не меняет результат операции, но валидная сессия не требуется.

## Идемпотентность

Повтор с тем же состоянием БД даёт тот же результат, кроме счётчика ограничения частоты.

## Основной сценарий

1. Проверить источник, тип и размер тела.
2. Нормализовать `email`.
3. Увеличить счётчик `auth_identify` для нормализованного адреса: 20 запросов на 900 секунд.
4. Найти локального пользователя по точному нормализованному значению.
5. Проверить существование хотя бы одного принадлежащего ему ключа доступа.
6. Вернуть результат.

## Альтернативные сценарии

Если подсистема ограничения частоты недоступна по любой причине, кроме уже достигнутого лимита, запрос продолжается, а технический отказ журналируется. Это намеренная fail-open ветка только данного предварительного определения.

## Ошибочные сценарии

| HTTP | Код | Условие | Тело |
|---:|---|---|---|
| 400 | `VALIDATION_ERROR` | невалидное тело или пустой нормализованный e-mail | стандартная ошибка BFF |
| 403 | `FORBIDDEN` | источник не совпал | `{"error":{"code":"FORBIDDEN","message":"Источник запроса не разрешён."}}` |
| 429 | `RATE_LIMITED` | превышено 20 запросов за 15 минут | стандартная ошибка BFF |
| 500 | `INTERNAL_ERROR` | ошибка локального хранилища или иной необработанный отказ | стандартная ошибка BFF |

В production стандартная ошибка имеет `{"error":{"code","message"}}`; в непроизводственном режиме добавляется `error.debug`.

## Логический результат

`303 See Other` to `/login?email=...` for a known identity or `/register?email=...` for an unknown identity. Rails renders the destination page.
## Побочные эффекты

Изменяется состояние ограничения частоты; создаются структурированные записи доступа. Пользовательские и внешние данные не меняются.

## Транзакционные требования

Атомарность между счётчиком и чтением пользователя не требуется. Чтение ключей ограничено проверкой существования одной записи.

## Наблюдаемость

Журналируются получение запроса, решение proxy, отправка BFF-ответа и отдельное предупреждение при fail-open счётчика без раскрытия полного e-mail: сохраняется только доменная часть.
