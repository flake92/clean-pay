# Получить совместимый профиль

## Идентификатор

`HTTP-039`

## Назначение

Сохранить старый transport-контракт профиля для существующего интерфейса/клиентов.

## Владелец

Модуль доступа и идентичности.

## Акторы

Вошедший пользователь.

## Предусловия

Пограничная политика пропускает session candidate; handler окончательно проверяет сессию.

## Логический входной контракт

Нет; тело/query не используются.

## Текущий транспорт

`GET /api/me`; session cookies.

## Правила валидации

Профиль читается из локального пользователя; Telegram bigint сериализуется строкой; Date сериализуется JSON-механизмом в ISO-8601, null сохраняется.

## Нормализация

Поля строго camelCase; BFF-оболочки `data` нет.

## Авторизация

Действующая access либо восстановимая refresh session. Нет сессии — handler `401 {"user":null}`; edge unauthenticated может раньше вернуть стандартную 401 envelope, поэтому эти две формы реально возможны в зависимости от cookie candidate.

## Идемпотентность

Чтение; refresh может ротировать cookie/токены.

## Основной сценарий

Прочитать текущего пользователя и вернуть совместимую проекцию.

## Альтернативные сценарии

Nullable e-mail, Telegram, имя, avatar и last login сохраняются null.

## Ошибочные сценарии

`401`; непредвиденная ошибка session/БД — framework 500, специализированной BFF-нормализации нет.

## Логический результат

`200 {"user":{"id":string,"email":string|null,"telegramId":string|null,"telegramUsername":string|null,"fullName":string|null,"photoUrl":string|null,"displayName":string|null,"emailVerified":boolean,"lastLoginAt":string|null}}`.

## Побочные эффекты

Возможный refresh/rotation и access log.

## Транзакционные требования

По контракту сессий.

## Наблюдаемость

Профиль целиком не журналируется.

## Источники

Доказательства находятся в `09-traceability/`; основной новый профиль — HTTP-004.

## Статус уверенности

`подтверждено`
