# Изменить пароль

## Идентификатор

`HTTP-006`

## Назначение

Изменить пароль Remnashop, затем отозвать все прежние локальные сессии и выдать текущему браузеру единственную новую сессию.

## Владелец

Модуль доступа и идентичности.

## Акторы

Полностью аутентифицированный пользователь с подтверждённым e-mail либо Telegram-идентичностью.

## Предусловия

Полная локальная сессия, разрешение политики e-mail, доступная внешняя связка token.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `current_password` | string | да | нет | строка | внешняя схема 1…256 | нет | доказательство текущего секрета |
| `new_password` | string | да | нет | строка | внешняя схема 8…256 | нет | новый секрет |

Дополнительные поля передаются Remnashop и закреплённой схемой игнорируются.

## Текущий транспорт

`PATCH /account/password`. Rails form scope `password[current_password,new_password]`; CSRF; secrets are passed only to Remnashop.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Общий JSON-контракт проверяет Clean Pay, полевые ограничения — Remnashop. Пароли не очищаются и не нормализуются.

## Нормализация

Нет; пробелы являются частью пароля.

## Авторизация

Proxy требует обычную полную сессию и подтверждённый e-mail для e-mail-only пользователя. Сервер дополнительно восстанавливает Remnashop token и сверяет внешний профиль/владельца.

## Идемпотентность

Нет. После успеха повтор со старым `current_password` должен завершиться ошибкой; повтор с новым паролем может изменить пароль снова.

## Основной сценарий

1. Авторизовать внешние токены.
2. Вызвать RS-006 с текущим access token.
3. После внешнего успеха в локальной транзакции заблокировать пользователя и текущую сессию.
4. Отозвать все активные локальные сессии, очистить в них внешние токены и сроки.
5. Создать новую сессию, сохранив метод, assurance, user-agent и ipHash текущей; записать новые внешние токены.
6. Установить новую пару локальных cookie и audit `password_changed`.

## Альтернативные сценарии

Если первая RS-006 вернула именно `CURRENT_PASSWORD_INVALID`, Clean Pay один раз вызывает RS-005 по refresh token, сохраняет обновлённые внешние токены и повторяет RS-006 ровно один раз. Этот повтор предназначен для устаревшего access token, хотя внешний код ошибки совпадает с неверным текущим паролем.

## Ошибочные сценарии

| HTTP | Код | Условие |
|---:|---|---|
| 400 | `VALIDATION_ERROR` | тело/поля не прошли схему |
| 401 | `CURRENT_PASSWORD_INVALID`/`UNAUTHORIZED` | пароль неверен после единственного refresh-повтора либо сессия уже неактивна |
| 403 | `FORBIDDEN`/`EMAIL_NOT_VERIFIED` | источник или политика доступа |
| 409 | `ACCOUNT_MERGE_REQUIRED`/`CONFLICT` | внешний владелец не согласован |
| 502 | `UPSTREAM_UNAVAILABLE`/`UPSTREAM_ERROR` | Remnashop недоступен/сломал контракт |
| 500 | `INTERNAL_ERROR` | локальная замена сессии или аудит |

Критический необратимый частичный успех: если пароль уже изменён вовне, но локальная replacement-транзакция не удалась, Clean Pay fail-closed отзывает все ещё активные локальные сессии пользователя и удаляет обе cookie текущего браузера. Изменение пароля не компенсируется.

## Логический результат

`303 See Other` to `/profile` with flash after replacing the local session family.
## Побочные эффекты

Изменение внешнего пароля; возможная ротация внешних токенов; массовый отзыв локальных сессий; создание новой; аудит.

## Транзакционные требования

Локальная замена выполняется одной БД-транзакцией с row locks и таймаутами транзакции по умолчанию конкретного вызова. Внешнее изменение предшествует ей и не атомарно с ней.

## Наблюдаемость

Этапы внешней авторизации, единственного retry, внешнего успеха, replacement и число отозванных сессий журналируются без паролей и token.
