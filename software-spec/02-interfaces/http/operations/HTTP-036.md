# Получить публичную готовность

## Идентификатор

`HTTP-036`

## Назначение

Сообщить наружу только агрегированную свежесть/готовность без раскрытия топологии и ошибок зависимостей.

## Владелец

Платформенный модуль.

## Акторы

Оркестратор, балансировщик, оператор.

## Предусловия

Нет; свежий снимок создаёт HTTP-037 либо планировщик readiness.

## Логический входной контракт

Нет.

## Текущий транспорт

`GET /health/readiness`; публичная машинная JSON readiness-проверка зависимостей.

## Правила валидации

Сначала читается Redis key `clean-pay:health:readiness:v1`, затем process-memory fallback. Снимок свеж, если дата parseable и возраст не более 90 000 мс.

## Нормализация

Только свежий snapshot со статусом `ok` становится `ok`; всё остальное fail-closed `degraded`.

## Авторизация

Нет.

## Идемпотентность

Чтение; не запускает проверки.

## Основной сценарий

Вернуть агрегат 200 со свежим `checkedAt`.

## Альтернативные сценарии

Нет/битый/старый snapshot, Redis недоступен и нет свежей memory copy — 503 degraded.

## Ошибочные сценарии

Управляемый degraded — не исключение, а 503. Необработанная ошибка handler/framework возможна отдельно.

## Логический результат

`200` при готовности или `503` при деградации; `application/json`, без секретов.

## Побочные эффекты

Redis GET и access log.

## Транзакционные требования

Нет.

## Наблюдаемость

Публичный ответ намеренно не содержит names/messages отдельных checks.

## Источники

Доказательства находятся в `09-traceability/`; operational policy — раздел 07.

## Статус уверенности

`требует повторной проверки после ADR-003`
