# Зарегистрироваться или возобновить регистрацию по e-mail

## Идентификатор

`HTTP-003`

## Назначение

Создать учётную запись Remnashop либо доказанно войти в уже существующую с тем же e-mail и паролем, создать локальную сессию и при необходимости инициировать отправку кода подтверждения.

## Владелец

Логический модуль доступа и идентичности.

## Акторы

Гость и форма регистрации.

## Предусловия

При включённом Turnstile нужен действующий token. Для ветки возобновления пароль должен совпасть с уже существующей внешней записью.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `email` | string | да | нет | e-mail | внешняя схема: trim/lower-case, ≤255, e-mail pattern | нет | владелец регистрации и ключ лимита |
| `password` | string | да | нет | строка | внешняя схема: 8…256 символов | нет | секрет новой записи/доказательство возобновления |
| `name` | string | нет | нет | строка | после trim 1…128 | отсутствует | имя пользователя |
| `referral_code` | string | нет | нет | строка | после trim 3…64; пустое превращается внешней схемой в отсутствие | отсутствует | реферальная атрибуция |
| `turnstileToken` | string | нет | нет | непрозрачный token | приоритетный вариант | отсутствует | антибот |
| `cf-turnstile-response` | string | нет | нет | непрозрачный token | fallback | отсутствует | антибот |

После удаления двух полей Turnstile остальные дополнительные поля уходят в Remnashop и игнорируются его закреплённой схемой.

## Текущий транспорт

`POST /account/registration`. Rails form scope `registration[email,password,turnstile_token]`; CSRF and Turnstile apply.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Общий разбор тела выполняет Clean Pay; полевую схему выполняет Remnashop. Turnstile и локальный rate limit проходят до создания/входа.

## Нормализация

Clean Pay не меняет поля до лимита и внешнего запроса. Remnashop нормализует e-mail, имя и referral согласно таблице.

## Авторизация

Публичная операция. При ветке существующего e-mail пароль проверяется отдельным внешним входом; сам факт конфликта e-mail не даёт доступа.

## Идемпотентность

Нет. Повтор может перейти из `register` в `login`, создаёт новую локальную сессию и может повторно запросить письмо с учётом 60-секундного cooldown.

## Основной сценарий

1. Проверить запрос и Turnstile.
2. Увеличить строгий счётчик `auth_register`: 5 на 900 секунд по входному e-mail.
3. Вызвать RS-001.
4. Получить внешний профиль и создать локального владельца/сессию.
5. Если внешний e-mail не подтверждён, вызвать RS-009 с этим адресом.
6. Записать audit и вернуть `201`.

## Альтернативные сценарии

- Только внешний `409`, чьё сообщение содержит `email already exists`, переключает алгоритм на RS-002 с теми же `email`/`password`. Успешный вход считается возобновлением регистрации и всё равно возвращает `201`.
- Если профиль уже подтверждён, письмо не запрашивается и поле `emailVerification` отсутствует.

## Ошибочные сценарии

| HTTP | Код | Условие |
|---:|---|---|
| 400 | `VALIDATION_ERROR` | тело или внешние поля недопустимы |
| 401 | `AUTH_FAILED` | e-mail существует, но пароль не подошёл при возобновлении |
| 403 | `FORBIDDEN` | источник или Turnstile отвергнут |
| 409 | `CONFLICT` | любой конфликт, кроме точного распознанного существующего e-mail, либо конфликт владельца |
| 429 | `RATE_LIMITED` | локальный/внешний предел или 60-секундный cooldown письма |
| 502 | `UPSTREAM_UNAVAILABLE`/`UPSTREAM_ERROR` | транспорт, таймаут 15 секунд, 5xx или недопустимый JSON Remnashop; сюда же может попасть отказ отправки письма после исчерпания разрешённых повторов |
| 503 | `UPSTREAM_UNAVAILABLE` | недоступность Turnstile |
| 500 | `INTERNAL_ERROR` | локальный отказ |

`504` операция не формирует.

Критический частичный успех: локальная сессия создаётся **до** запроса verification email. Если отправка письма завершается ошибкой, HTTP-ответ будет ошибочным, но учётная запись и сессия уже могут существовать, а cookie уже могут быть установлены через контекст ответа. Повтор должен безопасно пройти ветку возобновления, а не создавать второго владельца.

## Логический результат

`303 See Other` to `/register/verify-email` with Rails session cookies.
## Побочные эффекты

Внешняя учётная запись или внешний вход; письмо; локальный пользователь; полная сессия и cookie; зашифрованные внешние токены; rate-limit; аудит с flow `created` или `existing_email_login`.

## Транзакционные требования

Внешняя регистрация, локальная сессия и отправка письма не образуют одну транзакцию. Компенсирующего удаления внешней или локальной учётной записи при ошибке письма нет.

## Наблюдаемость

Фиксируются начало, Turnstile, лимит, внешняя ветка, отправка verification и итоговый audit. Пароль, токены и код письма не журналируются.

## Источники

Доказательства перечислены только в разделе `09-traceability/`.

## Статус уверенности

`требует повторной проверки после ADR-003`
