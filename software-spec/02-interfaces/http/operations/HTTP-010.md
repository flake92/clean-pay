# Начать регистрацию ключа доступа

## Идентификатор

`HTTP-010`

## Назначение

Создать одноразовый WebAuthn challenge и вернуть браузеру параметры регистрации ключа доступа.

## Владелец

Модуль доступа и идентичности.

## Акторы

Пользователь с полной либо ограниченной `BOOTSTRAP`-сессией и WebAuthn-браузер.

## Предусловия

Локальная сессия. Для `FULL`-сессии действует политика подтверждённого e-mail или Telegram; `BOOTSTRAP` освобождена от неё, поскольку ключ завершает bootstrap.

## Логический входной контракт

Полей нет. Вход — cookie сессии и заголовок `User-Agent` только для последующего именования ключа, не для этой операции.

## Текущий транспорт

`POST /api/bff/auth/passkey/register/options`; bodyless; доверенный origin; `Content-Type` не требуется; query игнорируется.

## Правила валидации

Сессия восстанавливается с обычной refresh rotation. RP origin берётся из точного публичного URL, RP ID — его hostname, RP name — текущее имя бренда.

## Нормализация

Имя WebAuthn-пользователя: локальный e-mail → Telegram username → Telegram ID → локальный user ID. Отображаемое имя: локальный `displayName` → `fullName` → выбранное имя пользователя.

## Авторизация

Разрешены `FULL` и `BOOTSTRAP`; отсутствующая сессия — 401. Proxy допускает bootstrap по всему префиксу passkey.

## Идемпотентность

Нет: каждый успех создаёт новый независимый challenge.

## Основной сценарий

1. Восстановить и проверить сессию.
2. Сгенерировать WebAuthn creation options.
3. Сохранить challenge типа `REGISTRATION`, владельца user ID и срок `now+5m`.
4. Вернуть options.

## Альтернативные сценарии

Параллельные challenges допустимы; каждый потребляется независимо ровно один раз.

## Ошибочные сценарии

`401 UNAUTHORIZED`, `403 EMAIL_NOT_VERIFIED`, `403 FORBIDDEN` origin, `500 INTERNAL_ERROR`. Тело не читается, поэтому 400/413/415 для этой операции не возникают.

## Логический результат

`200 {"data":<PublicKeyCredentialCreationOptionsJSON>}`. Неизменяемые параметры: `rp.id=<public hostname>`, `rp.name=<brand>`, user handle — UTF-8 локального user ID в base64url, `timeout=120000`, `attestation="none"`, `authenticatorSelection.residentKey="preferred"`, `userVerification="required"`. `excludeCredentials` намеренно не задаётся. Алгоритмы и дополнительные стандартные поля формирует совместимая WebAuthn-библиотека.

## Побочные эффекты

Новая строка одноразового challenge; возможно обновление cookie при refresh rotation.

## Транзакционные требования

Создание одного challenge атомарно; одновременные challenges не конфликтуют.

## Наблюдаемость

Общие журналы HTTP/сессии; challenge как секрет не должен попадать в эксплуатационный журнал.

## Источники

Доказательства находятся в `09-traceability/`.

## Статус уверенности

`подтверждено`
