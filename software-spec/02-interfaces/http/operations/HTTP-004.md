# Получить текущий профиль аутентификации

## Идентификатор

`HTTP-004`

## Назначение

Вернуть профиль текущего владельца, при возможности проверить его во внешней системе и согласовать уже подтверждённый e-mail.

## Владелец

Модуль доступа и идентичности.

## Акторы

Вошедший пользователь и клиентский интерфейс.

## Предусловия

Действующая полная или разрешённая для подтверждения e-mail локальная сессия. Proxy допускает этот путь для неподтверждённого e-mail.

## Логический входной контракт

Полей нет. Query и тело не используются; присланное GET-тело обработчик не читает. Вход — cookie `clean_pay_access` и/или `clean_pay_refresh`.

## Текущий транспорт

`GET /account/session`. Bodyless Rails resource read using session cookies.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Access проверяется по HMAC, сроку и активной записи сессии. При отсутствии/ошибке/истечении access выполняется атомарная попытка refresh rotation. Предыдущий refresh допускается 10 секунд и возвращает того же преемника; reuse за пределом grace отзывает сессию.

## Нормализация

Локальные Telegram ID сериализуются строкой. Поля внешнего профиля сохраняют wire-имена; дополнительно формируются camelCase-поля совместимости.

## Авторизация

Требуется локальная сессия. Неподтверждённый e-mail разрешён для этой операции; bootstrap-сессия проходит только по общему набору passkey-путей и сюда proxy её не допускает.

## Идемпотентность

Логически чтение, но может ротировать refresh token, обновить access-cookie и зафиксировать подтверждение e-mail. Повтор безопасен по итоговому состоянию, но не является чистым чтением транспорта.

## Основной сценарий

1. Восстановить локальную сессию.
2. Если нет внешних токенов, внешнего owner ID и Telegram ID, вернуть локальный профиль.
3. Иначе получить/обновить/восстановить токены Remnashop и вызвать RS-007.
4. Если внешний e-mail совпадает с локальным, уже подтверждён вовне, а локально ещё нет либо установлен `authPending`, зафиксировать локальное подтверждение и очистить pending-поля.
5. Вернуть объединённое представление.

## Альтернативные сценарии

- Невозможность получить внешние токены с логическим `EMAIL_REQUIRED` преобразуется в локальный профиль, а не ошибку.
- Telegram-сессия может восстановить внешние токены через доказанную Telegram-идентичность.
- Refresh access token Remnashop выполняется до `/auth/me`, поэтому заведомо истёкший token не используется.

## Ошибочные сценарии

| HTTP | Код | Условие |
|---:|---|---|
| 401 | `UNAUTHORIZED` | локальная сессия не восстановлена или внешняя авторизация отвергнута |
| 409 | `ACCOUNT_MERGE_REQUIRED`/`CONFLICT` | идентичность противоречит единственному владельцу |
| 502 | `UPSTREAM_UNAVAILABLE`/`UPSTREAM_ERROR` | внешняя сеть, таймаут 15 секунд, 5xx или невалидный ответ |
| 500 | `INTERNAL_ERROR` | локальное хранилище/согласование |

Proxy может вернуть `401` раньше обработчика и удалить обе cookie. `503`/`504` обработчик не формирует.

## Логический результат

`303 See Other` to the server-rendered `/profile`; a guest is redirected to `/login` with flash.
## Побочные эффекты

Возможны refresh rotation, очистка скомпрометированной сессии, восстановление внешних токенов, синхронизация подтверждения e-mail и технический журнал.

## Транзакционные требования

Refresh rotation атомарно блокирует сессию. Синхронизация e-mail — отдельное локальное изменение после внешнего чтения; распределённой транзакции нет.

## Наблюдаемость

Фиксируются источник сессии, наличие токенов, выбранная локальная/внешняя ветка и факт согласования без раскрытия токенов.

## Источники

Доказательства перечислены в `09-traceability/`.

## Статус уверенности

`требует повторной проверки после ADR-003`
