# Войти по e-mail и паролю

## Идентификатор

`HTTP-002`

## Назначение

Проверить антибот-доказательство, аутентифицировать пользователя в Remnashop, согласовать локального владельца и создать локальную веб-сессию.

## Владелец

Логический модуль доступа и идентичности.

## Акторы

Гость и браузерная форма входа.

## Предусловия

Пользователь располагает e-mail и паролем Remnashop. При включённой антибот-защите нужен действующий Turnstile token.

## Логический входной контракт

| Поле | Тип | Обязательно | Nullable | Формат | Ограничения | По умолчанию | Назначение |
|---|---|---:|---:|---|---|---|---|
| `email` | string | да | нет | e-mail | внешняя схема: trim, lower-case, не более 255, шаблон e-mail | нет | имя входа и ключ лимита |
| `password` | string | да | нет | строка | внешняя схема: 1…256 символов | нет | секрет входа |
| `turnstileToken` | string | нет | нет | непрозрачный token | используется первым, если не `null`/`undefined` | отсутствует | антибот-доказательство |
| `cf-turnstile-response` | string | нет | нет | непрозрачный token | совместимый fallback | отсутствует | антибот-доказательство |

Поля Turnstile удаляются перед внешним запросом. Все остальные дополнительные form-поля передаются Remnashop; закреплённая внешняя схема их игнорирует. Локальная TypeScript-форма не является runtime-валидацией.

## Текущий транспорт

`POST /account/session`. Rails form scope `session[email,password,turnstile_token]`; passwords remain byte-for-byte form values and are never stored locally.

ADR-003 заменяет исторический BFF/JSON transport этой операции.

Коды ошибок в нижележащем историческом анализе теперь являются доменными классификациями: браузеру Rails рендерит form errors/flash либо выполняет безопасный redirect; BFF envelope не возвращается.
## Правила валидации

Общий разбор проверяет только JSON-объект и размер. Типы, e-mail и длины проверяет Remnashop. При включённом Turnstile отсутствие/отказ токена завершается до внешней аутентификации.

## Нормализация

Clean Pay не нормализует `email` и `password` перед лимитом и внешним вызовом; нормализацию e-mail выполняет Remnashop. Поэтому регистр и внешние пробелы входят в локальную идентичность rate-limit так, как пришли, хотя внешний сервис может считать адреса одинаковыми.

## Авторизация

Публичная операция. Успех создаёт новую полную локальную сессию метода `EMAIL`.

## Идемпотентность

Неидемпотентна: каждый успешный повтор создаёт новую запись сессии и новую пару cookie. Существующие сессии этим входом не отзываются.

## Основной сценарий

1. Проверить браузерный источник, JSON media type и тело.
2. Извлечь Turnstile token с приоритетом `turnstileToken` над `cf-turnstile-response` и проверить его.
3. Увеличить строгий счётчик `auth_login`: 5 попыток на 900 секунд для входного значения e-mail.
4. Вызвать RS-002 и получить две auth-cookie Remnashop.
5. Получить проверенный внешний профиль, согласовать ровно одного локального владельца и сохранить зашифрованные внешние токены в локальной сессии.
6. Создать access token на 15 минут и случайный refresh token на 30 дней.
7. Записать успешный аудит и вернуть профиль.

## Альтернативные сценарии

При существующем локальном пользователе его профиль и внешняя связь согласуются по правилам единственного владельца. Лимитер для этой операции fail-closed: его технический отказ не разрешает вход.

## Ошибочные сценарии

| HTTP | Код | Условие |
|---:|---|---|
| 400 | `VALIDATION_ERROR` | тело или внешняя схема недопустимы |
| 401 | `AUTH_FAILED` | Remnashop отверг данные входа |
| 403 | `FORBIDDEN` | источник или Turnstile не прошли |
| 409 | `CONFLICT` либо ошибка владельца | вход противоречит существующему владению |
| 429 | `RATE_LIMITED` | превышен локальный или внешний лимит |
| 502 | `UPSTREAM_UNAVAILABLE`/`UPSTREAM_ERROR` | сетевой таймаут 15 секунд, транспорт, 5xx или недопустимый ответ Remnashop |
| 503 | `UPSTREAM_UNAVAILABLE` | только недоступность проверки Turnstile |
| 500 | `INTERNAL_ERROR` | локальная ошибка согласования, хранилища или создания сессии |

`504` этот интерфейс не формирует. Любой неуспех дополнительно создаёт audit action `auth_login_failed`; e-mail может присутствовать в его metadata и потому является чувствительным журналируемым значением.

## Логический результат

`303 See Other` to `/cabinet`, `/verify-email`, or `/passkey/setup`; Rails session cookies are set before redirect.
## Побочные эффекты

Счётчик попыток, локальный пользователь/согласование профиля, новая сессия, зашифрованные токены Remnashop, audit success/failure и технические журналы. Успешная внешняя аутентификация при последующей локальной ошибке не создаёт откатываемой операции во внешней системе.

## Транзакционные требования

Согласование пользователя выполняет свои локальные транзакционные проверки; внешний вход и локальная фиксация не образуют распределённую транзакцию.

## Наблюдаемость

Логи фиксируют этапы Turnstile, лимита, внешнего входа, согласования и сессии без записи пароля или токенов. Audit различает `auth_login_success` и `auth_login_failed`.
