# Получить список ключей доступа

## Идентификатор

`HTTP-014`

## Назначение

Показать все ключи текущего владельца для управления ими.

## Владелец

Модуль доступа и идентичности.

## Акторы

Пользователь профиля/связывания аккаунта.

## Предусловия

Полная сессия и политика подтверждённого e-mail либо Telegram.

## Логический входной контракт

Полей нет; GET-body/query не читаются; cookie сессии обязательны.

## Текущий транспорт

`GET /api/bff/auth/passkey/credentials`.

## Правила валидации

Session assurance обязана быть точной `FULL`; затем проверяется email/Telegram policy.

## Нормализация

Date сериализуются JSON как ISO date-time; bigint counter и public key не выводятся.

## Авторизация

Только собственные записи по локальному user ID.

## Идемпотентность

Логическое чтение; возможна общая refresh rotation.

## Основной сценарий

Найти все credentials владельца с сортировкой `createdAt ASC` и вернуть безопасную проекцию.

## Альтернативные сценарии

Нет ключей → пустой массив.

## Ошибочные сценарии

`401 UNAUTHORIZED`, `403 EMAIL_NOT_VERIFIED`, `500 INTERNAL_ERROR`.

## Логический результат

`200 {"data":{"credentials":[...]}}`, где каждый элемент содержит обязательные `id`, `credentialId`, `name:string|null`, `transports:string[]`, `deviceType:string|null`, `backedUp:boolean`, `lastUsedAt:date-time|null`, `createdAt:date-time`. Порядок — от старого к новому.

## Побочные эффекты

Только возможная refresh rotation и журналы.

## Транзакционные требования

Обычный согласованный снимок одного SELECT; блокировки не требуются.

## Наблюдаемость

Credential public keys и counters не выводятся/не журналируются.

## Источники

Доказательства находятся в `09-traceability/`.

## Статус уверенности

`подтверждено`
