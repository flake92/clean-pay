# Ограничения времени и нагрузки

| Операция/граница | Бюджет |
|---|---:|
| обычный запрос Remnashop | 15 секунд |
| recovery, Telegram, Turnstile, Remnawave | 10 секунд |
| одна readiness-зависимость | 5 секунд |
| весь detailed readiness | 8 секунд |
| worker HTTP-вызов | 45 секунд |
| один reconciliation/history элемент | 12 секунд |
| ожидание блокировки ротации/передачи токенов | 5 секунд |
| транзакция ротации | 15 секунд |
| транзакция передачи токенов | 20 секунд |

Интерактивная синхронизация запрашивает у upstream не более 100 записей, но UI-ответ выдаёт не более 20. Worker-пакет ограничен 1..100 элементами, по умолчанию 10. Backfill продвигается по одной cursor-странице на захват. Readiness объединяет конкурентные одинаковые проверки и кэширует итог; frontend объединяет параллельные чтения одних panel/menu данных и использует ограниченный backoff при опросе платежа.

Числовые throughput, latency percentile и SLA исходной системой не заданы и не должны выдумываться. Приёмка проверяет соблюдение тайм-аутов, ограниченность пакетов, отсутствие неограниченного роста памяти и то, что медленная интеграция не удерживает долгую транзакцию БД.
