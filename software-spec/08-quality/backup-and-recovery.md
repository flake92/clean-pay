# Резервное копирование и восстановление

## Объём резервной копии

Авторитетен полный PostgreSQL: схема, таблицы, ограничения и журнал применённых миграций. Redis полезно сохранять для плавного восстановления лимитов, но его readiness/rate-кэш может быть построен заново. Вместе с бэкапом безопасно и отдельно хранят версию образа, параметры конфигурации без значений секретов и идентификатор ключа `WEB_REFRESH_SECRET`.

## Перед релизом/миграцией

1. Снять согласованную резервную копию базы.
2. Восстановить её в изолированный PostgreSQL той же major-версии.
3. Проверить количество строк всех 15 моделей, внешние ключи и уникальности.
4. Запустить новую версию, применить ожидающие миграции и выполнить внутреннюю readiness.
5. Проверить выборочно вход, чтение истории, `OUTCOME_UNKNOWN`, merge и worker claims без внешнего платёжного эффекта.

## Восстановление production

Остановить writers, развернуть чистую базу, восстановить backup, сопоставить версию схемы, применить только ожидающие миграции, затем запустить приложение и workers. Проверить readiness, очереди неизвестных/ручных платежей, поколения синхронизации и свежесть heartbeat. Возврат трафика разрешён только после проверки.

Ключ шифрования внешних token bundles должен соответствовать данным: его потеря делает bundles нечитаемыми и требует повторного входа/привязки. Смена JWT-секрета инвалидирует только короткие access cookies. Автоматизированного механизма бэкапа приложение не предоставляет; наличие внешнего расписания и успешной учебной реставрации — обязательная эксплуатационная приёмка.
