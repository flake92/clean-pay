# Сценарии платежей

## PAY-SC-001. Подготовка покупки

PAGE-009 загружает предложения HTTP-023 и сохраняет только выбранные идентификаторы. PAGE-010 перед подтверждением повторно получает актуальное предложение. Сумма, валюта, тариф, длительность, gateway и версия предложения берутся из подтверждённого сервером предложения, а не из доверенного браузерного текста.

## PAY-SC-002. Покупка

1. Браузер создаёт UUID-ключ повторяемости и вызывает HTTP-024.
2. Сервер связывает `(пользователь, ключ)` с отпечатком команды, фиксирует `DISPATCHING` до внешнего вызова и формирует собственный `return_url`.
3. Синхронный успех сохраняет платёж и возвращает `200`; принятая обработка возвращает `202` и интервал повтора; ручная проверка — `409` с `manual_required`.
4. `payment_url` открывается top-level переходом. Его отсутствие обрабатывается по состоянию payload, а не угадывается по HTTP-коду.

## PAY-SC-003. Продление

HTTP-025 повторяет защиту PAY-SC-002, но принимает только предложение продления действующей подписки. Изменённое или устаревшее предложение отклоняется до отправки. Локальная дата окончания не меняется до подтверждённого результата.

## PAY-SC-004. Повтор и неопределённый исход

- Тот же ключ и отпечаток возвращают ранее сохранённый результат с признаком replay.
- Тот же ключ с другим отпечатком даёт конфликт без внешнего вызова.
- Ошибка до dispatch допускает окончательный отказ; таймаут или разрыв после возможной отправки переводит операцию в `OUTCOME_UNKNOWN`.
- `OUTCOME_UNKNOWN` нельзя повторно оплачивать вслепую: worker или HTTP-038 выполняет сверку по поддерживаемому capability-контракту.
- После исчерпания безопасных попыток состояние становится `MANUAL_REQUIRED` и содержит операторское действие.

## PAY-SC-005. История и возврат провайдера

1. HTTP-031 синхронизирует внешнюю историю страницами под блокировкой внешнего владельца и не переписывает чужие платежи.
2. HTTP-032 принимает локальную ссылку операции/платежа, проверяет их принадлежность и возвращает локальный окончательный результат даже при временной недоступности внешней системы.
3. PAGE-012—PAGE-014 считают URL `success|fail|pending` только подсказкой; фактический статус всегда берётся из HTTP-032.
4. Retryable состояние опрашивается ограниченно с `retry_after_seconds`; закрытие страницы прекращает опрос и никогда не создаёт новую покупку.

## PAY-SC-006. Смена владельца

Перед объединением аккаунтов payment-owner fence блокирует новые неоднозначные dispatch. Операции с доказанно тем же внешним владельцем переносятся; операция с иным владельцем становится `MANUAL_REQUIRED`. Ни одна ветка не изменяет уже выполненный платёж задним числом.

## Проверки

Для покупки и продления обязательна таблица тестов по каждому статусу, заголовкам `idempotency-replayed`, `x-payment-operation-id`, `retry-after`, одинаковому/разному отпечатку, конкурентным запросам, внешним 4xx/5xx/таймаутам, падению БД до/после dispatch, изменению предложения и восстановлению после рестарта.
