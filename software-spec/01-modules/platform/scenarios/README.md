# Сценарии платформы и эксплуатации

## PLT-SC-001. Жизнь и готовность

- HTTP-034 отвечает базовым здоровьем процесса без внешних обращений; HTTP-035 — отдельная liveness-граница оркестратора.
- HTTP-036 возвращает только безопасное публичное `ready|degraded` по свежему снимку.
- HTTP-037 с правильным служебным секретом проверяет PostgreSQL, Redis, Remnashop, Remnawave, опциональный Mailpit и сердцебиения worker; неверный секрет маскируется под 404.
- HTTP-033 отдельно возвращает конфигурированные контакты поддержки.

## PLT-SC-002. Очистка данных

Периодический исполнитель каждые 300—86400 секунд удаляет только данные, подпадающие под отдельные сроки хранения: истёкшие одноразовые состояния, старые отозванные сессии, информационный/защитный аудит и события лимитов. Платежи этой задачей не удаляются. Ошибка одной итерации журналируется; следующая итерация выполняется после интервала.

## PLT-SC-003. Сверка платежей

При включённой функции worker вызывает HTTP-038 с секретом, ограниченным batch и арендой. Один объект имеет одного владельца аренды; просроченная аренда возвращается в работу. Сбой приводит к следующей ограниченной попытке, а необратимая неоднозначность — к ручной проверке.

## PLT-SC-004. Запуск и миграции

До приёма запросов валидируются конфигурация, секреты, origin и сеть, затем доступность PostgreSQL/Redis и последовательное применение ожидающих миграций. Ошибка прекращает старт; рабочая БД не пересоздаётся. После старта readiness остаётся отрицательным до свежего успешного снимка.

## PLT-SC-005. PWA и offline

HTTP-044 обслуживает фиксированный service worker. Он кэширует только публичную оболочку/статику и offline fallback; API, персональные страницы и session-dependent ответы не кэшируются. Новая версия меняет cache namespace и удаляет только принадлежащие приложению старые кэши.

## PLT-SC-006. Поддержка и оболочка

HTTP-033 возвращает включённые каналы поддержки. PAGE-017 показывает e-mail, Telegram и FAQ только при корректной конфигурации; при отсутствии каналов отображает точную пустую копию. Внешние ссылки используют безопасные системные переходы. HTTP-040 остаётся отдельным legacy-выходом и не подменяется BFF-операцией HTTP-005.

## PLT-SC-007. Отказ и восстановление процесса

SIGTERM прекращает приём новой работы, завершает текущую в ограниченный срок и оставляет аренды восстанавливаемыми. После рестарта состояние восстанавливается из PostgreSQL/Redis, а не из памяти процесса. Процессный кэш может ускорять чтение, но не становится единственным источником устойчивого состояния.

## Проверки

Обязательны invalid-config, dependency-down, stale readiness cache, неверный internal secret, просроченное heartbeat, частичный сбой очистки, конкурирующие worker, рестарт во время аренды, offline/PWA update и отсутствие утечки секретов в ответах/логах.
