# Сценарии доступа и идентичности

Все сценарии сохраняют точные транспортные контракты карточек HTTP-001—HTTP-020, HTTP-039 и HTTP-041—HTTP-043. Названия внутренних классов и фреймворка не являются частью сценария.

## IAM-SC-001. Определение способа входа

1. Гость вводит e-mail на PAGE-002.
2. Система нормализует адрес и вызывает HTTP-001.
3. Для существующего адреса UI открывает ввод пароля; для отсутствующего — предлагает регистрацию; ошибки проверки не создают пользователя.
4. Ограничитель частоты работает fail-open только при технической недоступности своего хранилища, но не при достигнутом лимите.

## IAM-SC-002. Вход по e-mail и ротация сессии

1. Пользователь отправляет адрес, пароль и при включённой защите Turnstile-token через HTTP-002.
2. После внешней проверки создаётся или связывается локальный владелец и выдаётся семейство защищённых cookie.
3. Неподтверждённый e-mail переводит пользователя на PAGE-005; полный пользователь — в кабинет; bootstrap-сессия — на PAGE-007.
4. При истёкшей короткой части общий слой прозрачно вращает длинную часть. Повтор предыдущей длинной части в допустимом окне возвращает того же преемника; поздний повтор отзывает всё семейство.
5. HTTP-005 отзывает все активные локальные сессии пользователя и очищает cookie независимо от доступности внешней системы.

## IAM-SC-003. Регистрация и подтверждение почты

1. HTTP-003 регистрирует адрес либо, при точно распознанном существующем адресе и верном пароле, продолжает как вход.
2. Система запрашивает отправку шестизначного кода через Remnashop → SMTP; код и токены не попадают в журнал.
3. HTTP-008 одноразово подтверждает код. Неверный, истёкший или повторный код не подтверждает владение.
4. Подтверждение фиксируется до необязательной синхронизации владельцев. Если синхронизация не завершилась, ответ остаётся успешным с `account_sync_pending=true`.
5. Повторный запрос письма подчиняется отдельному rate limit и внешнему cooldown.

## IAM-SC-004. Telegram OIDC

1. HTTP-041 создаёт state, nonce и PKCE verifier, сохраняет одноразовое состояние и перенаправляет на OIDC authorization endpoint.
2. HTTP-042 принимает OIDC callback, атомарно потребляет состояние, обменивает code, проверяет подпись JWT, issuer, audience и nonce. HTTP-043 обслуживает popup/Login Widget callback по его отдельному точному контракту.
3. Подтверждённая Telegram-личность связывается с единственным локальным владельцем; неоднозначное совпадение не разрешается автоматически.
4. Для подтверждённого внешнего e-mail создаётся полная сессия; для нового e-mail — ограниченная ветка подтверждения; для bootstrap — PAGE-007.
5. Ошибка провайдера или проверки возвращает пользователя на вход с публичным кодом, не раскрывая token/state.

## IAM-SC-005. Telegram WebApp

1. PAGE-006 получает подписанные `initData` из Telegram WebApp SDK и вызывает HTTP-016.
2. Система передаёт строку во внешний Telegram-вход Remnashop; браузерская распаковка не считается доказательством личности.
3. Успех создаёт ту же модель владельца и сессии, что OIDC; повтор и конфликт владельцев проходят общие правила объединения.

## IAM-SC-006. Ключ доступа WebAuthn

1. HTTP-010/HTTP-012 создают короткоживущий одноразовый challenge для регистрации/входа.
2. HTTP-011/HTTP-013 атомарно потребляют challenge и проверяют origin, RP ID, подпись, credential и счётчик.
3. Регистрация существующего credential разрешена только тому же владельцу с тем же открытым ключом.
4. Конкурентное продвижение ненулевого счётчика имеет одного победителя; проигравший не получает сессию.
5. HTTP-014 перечисляет ключи, HTTP-015 удаляет только принадлежащий ключ и никогда не удаляет последний.
6. HTTP-016 завершает bootstrap без создания ключа, если политика допускает пропуск.

## IAM-SC-007. Изменение профиля и пароля

1. HTTP-009 меняет pending e-mail, немедленно снимает локальное подтверждение и затем отправляет письмо. Ошибка письма не откатывает смену адреса.
2. HTTP-006 меняет пароль после внешней проверки, отзывает старые сессии и выдаёт текущему браузеру новую.
3. HTTP-004 всегда возвращает профиль в совместимом представлении и может завершить локальную синхронизацию подтверждённого e-mail.

## IAM-SC-008. Привязка и объединение аккаунтов

1. HTTP-020 проверяет пароль целевого e-mail-владельца и либо связывает Telegram сразу, либо запускает подтверждение почты.
2. При конфликте владельцев создаётся одноразовое подтверждение, отображаемое PAGE-016; HTTP-017 читает его состояние.
3. HTTP-018 повторно проверяет доказательства, внешнее dry-run объединение, подписки и отсутствие опасной платёжной операции.
4. Под блокировкой владельца переносятся идентичности, ключи, платежи и допустимые данные; конкурирующие сессии отзываются.
5. Частичный или неоднозначный внешний результат не маскируется: автоматическое объединение прекращается и требуется безопасное продолжение либо оператор.

## Критерий сценарного покрытия

Для каждого сценария обязательны проверки: штатный успех, неверный ввод, отсутствие/`null`, лишние поля согласно карточке операции, отсутствие прав, rate limit, повтор, конкурентный вызов, внешняя ошибка/таймаут, локальная ошибка и проверка всех cookie/редиректов/изменений данных.
