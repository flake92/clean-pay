# Сценарии

## Вход по почте

1. Пользователь вводит адрес.
2. Система сообщает интерфейсу, существует ли локальный пользователь и есть ли ключ доступа.
3. Для существующего пользователя запрашивается пароль или предлагается быстрый вход.
4. Для нового пользователя показываются пароль и повтор пароля.
5. После успешной проверки создаётся сессия; при необходимости пользователь направляется на подтверждение почты или настройку ключа доступа.

## Telegram

1. Система создаёт одноразовые состояние, проверочное значение и доказательство возврата.
2. Пользователь проходит Telegram.
3. Возврат сопоставляется с началом и поглощает одноразовое состояние.
4. Telegram-идентичность связывается с внешней и локальной учётной записью.
5. Если обнаружены два владельца, создаётся явное подтверждение объединения; до решения данные не смешиваются.

## Ключ доступа

1. Система выдаёт одноразовые параметры регистрации или входа.
2. Браузер получает подтверждение пользователя на устройстве.
3. Система проверяет происхождение, назначение, одноразовый вызов и криптографический ответ.
4. При регистрации ключ сохраняется; при входе создаётся полная сессия.

## Обновление сессии

1. При отсутствии действующей короткой части браузер предъявляет долгую часть.
2. Система блокирует семейство, поглощает предъявленное значение и создаёт одного преемника.
3. Параллельный запрос в коротком окне получает того же преемника.
4. Более поздний повтор считается компрометацией и отзывает сессию.
