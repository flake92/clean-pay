# Главный промпт реализации

Реализуй Clean Pay как Ruby-монолит по каталогу `software-spec/`, не обращаясь к прежнему исходному приложению. Сначала полностью прочитай `00-system/`, `99-llm/immutable-contracts.md`, `99-llm/implementation-order.md`, затем нормативные разделы `01`—`08` и принятые решения `10-decisions/accepted/`.

Правила работы:

- воспринимай приложение как чёрный ящик с входами, состояниями и выходами;
- не ставь продукту «диагноз» по устройству прежнего стека и не копируй случайную внутреннюю архитектуру;
- создай contract-тесты до реализации каждого интерфейса;
- сохраняй точные wire identifiers и русскую UI-копию;
- не объединяй похожие операции и не упрощай error/partial-success/retry ветки;
- используй PostgreSQL и Redis с описанной семантикой; внешние сервисы не встраивай в монолит;
- сохраняй test/spec-контейнеры и подключай новый app-контейнер к их существующим контрактам;
- не выполняй реальную оплату и не обращайся к production-системам в проверках;
- после каждого логического модуля прогоняй его contract/integration/concurrency tests;
- UI принимай по растровым эталонам, а не по субъективному сходству;
- любое неустановленное решение останови и оформи как вопрос, не угадывай.

Финальный результат принимается только по `99-llm/verification-prompt.md` и всем требованиям `08-quality/`. Наличие работающего happy path не является завершением.
