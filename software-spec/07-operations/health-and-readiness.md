# Жизнеспособность и готовность

## Уровни проверки

- Базовая жизнеспособность отвечает HTTP 200, если процесс принимает HTTP; она не вызывает зависимости.
- Подробная готовность параллельно проверяет PostgreSQL (`SELECT 1`), Redis (`PING`), публичные тарифы Remnashop и Telegram JWKS. Mailpit и метаданные Remnawave проверяются только когда соответствующие интеграции включены конфигурацией.
- На одну зависимость даётся 5 секунд, на весь набор — 8 секунд. Любая обязательная неуспешная проверка даёт итог `degraded`.

## Кэш

Итог хранится в памяти процесса и, при доступности, в Redis под ключом `clean-pay:health:readiness:v1` с TTL 120 секунд. Публичный снимок считается устаревшим через 90 секунд. Статус `ok` допустим только для свежего кэшированного успешного результата; отсутствие кэша либо устаревший снимок не маскируется как успех.

## Публичный и внутренний ответ

Публичная операция раскрывает только общий статус, время проверки, признак устаревания, имя сервиса и версию. Состояния отдельных зависимостей доступны только внутренней операции с секретом. Никакой ответ не раскрывает URL с credentials, токены, тексты внутренних исключений или конфигурацию.

## Контейнерная проверка

Контейнер вызывает внутреннюю готовность каждые 15 секунд с тайм-аутом 12 секунд, стартовой льготой 30 секунд и максимум 20 неуспешными попытками до признания unhealthy. Для worker дополнительно проверяется свежесть его heartbeat. Готовность — сигнал маршрутизации/оператора, а не гарантия успеха следующей бизнес-транзакции.
