# Среда выполнения и развёртывание

## Рабочая топология

Обязательные сервисы: приложение, worker очистки, PostgreSQL 17 и Redis 7. Worker сверки/истории запускается отдельным профилем, когда включён. Брокера сообщений нет. База и Redis используют именованные постоянные volumes.

Приложение слушает `0.0.0.0:4000` внутри контейнера; host-публикация по умолчанию `127.0.0.1:4000`, чтобы прямой внешний доступ шёл через reverse proxy. Приложение подключается к внутренним сетям базы/Redis и к заранее созданной внешней edge-сети `${CLEAN_PAY_EDGE_NETWORK:-remnawave-network}` для интеграций. Отсутствующая edge-сеть является ошибкой предварительной проверки, а не поводом создать неизвестную сеть молча.

## Старт

1. Разобрать и строго проверить production-конфигурацию, секреты, origin и сетевое владение.
2. Дождаться PostgreSQL/Redis согласно readiness зависимостей.
3. Если включено применение миграций, применить все ожидающие шаги; при ошибке прекратить запуск.
4. Запустить HTTP-процесс на порту 4000.
5. Отвечать готовностью только после подробной проверки зависимостей.

Язык и web-фреймворк новой реализации не фиксированы. Обязательны наблюдаемое поведение, маршруты, cookies, процессы запуска и сигналы из этой спецификации, а не текущий runtime.

## Политика перезапуска

Приложение, очистка, PostgreSQL и Redis перезапускаются `unless-stopped`. Worker сверки использует `on-failure` с конечным ограничением оркестратора. `SIGTERM` должен прекращать приём новых запросов/пакетов, дать текущей операции ограниченное время и завершить процесс без порчи аренды.

## Сборка и конфигурация

Образ приложения неизменяемый и не содержит production `.env`. Конфигурация поступает во время запуска. Debug-вариант развёртывания отделён и не должен ослаблять production security headers/secret validation случайно. Точные сервисные порты и тестовые контейнеры перечислены в `04-integrations/mock-services.md` и `02-interfaces/configuration.md`.

## Изоляция процессов сборки и разработки

Два процесса старой реализации, использующие один рабочий каталог и один каталог результата сборки, способны повредить или подменить артефакты друг друга. Наблюдалась ситуация, когда production-процесс и dev-процесс одновременно использовали общий результат сборки: health начал отвечать 500, хотя зависимости были исправны. Для любой новой реализации действует нейтральное правило:

- production, development, test и visual-render процессы используют раздельные каталоги сборки/cache либо разные контейнерные файловые системы;
- порт приложения проверяется до запуска runner; занятый порт является ошибкой предварительной проверки, а не причиной запустить второй процесс поверх первого;
- entrypoint не должен выбирать альтернативный корень маршрутов из-за пустого/исторического каталога;
- runner завершает только запущенный им процесс и не останавливает общие mock/spec-сервисы или volumes без явного флага сброса;
- CSP development-режима должен позволять механизм разработки только в dev-профиле; production CSP нельзя механически использовать там, где framework требует динамическую загрузку, и наоборот dev-ослабление нельзя переносить в production.

Эти правила не закрепляют Next.js: они фиксируют обнаруженный класс эксплуатационного отказа, который Ruby prestage тоже обязан исключить.

## Наблюдаемость вне HTTP-контекста

Доменный аудит должен принимать явный контекст операции и уметь записывать техническое событие из worker, консольной команды и конкурентного теста без обращения к request-local API. Отсутствие HTTP-заголовков допустимо и представляется `null`/отсутствующим полем; оно не должно приводить к потере самого security-аудита. Сбой дополнительного аудита не отменяет уже принятую защищающую транзакцию, но создаёт отдельное событие `audit_write_failed` и сигнал мониторинга.
