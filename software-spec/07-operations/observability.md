# Аудит, логирование и мониторинг

## Структурированный журнал

Каждая строка — JSON с временем, уровнем, стабильным именем события, категорией, источником, request/correlation ID и очищенными полями контекста. Обязательные семейства: HTTP получен/решение/завершён; внешний запрос отправлен/получен/ошибка; решение аутентификации; доменная мутация; аудит; запуск/остановка/пакет фонового исполнителя.

Путь журналируется без query string. Рекурсивная редактировка заменяет значения ключей `password`, `token`, `cookie`, `authorization`, `code`, `secret` и смысловых псевдонимов на маркер. Неизвестное исключение журналируется с безопасным классом/кодом; production-ответ не наследует его текст.

## Долговечный аудит

Запись содержит действие, `INFO|WARN|ERROR`, необязательного пользователя, ключевой отпечаток IP, очищенные JSON-метаданные и время. Для мутаций фиксируются успех и ошибка. Сбой записи аудита журналируется отдельно, но обычно не откатывает уже успешную бизнес-транзакцию.

## Операционные сигналы

Входы мониторинга: публичная/внутренняя готовность, container health, возраст heartbeat обоих workers, частота стабильных событий ошибок и состояние volumes. Экспортера метрик и distributed tracing в текущем контракте нет; новая реализация может добавить их, не заменяя стабильные JSON-события.

## Ротация

Для контейнера приложения JSON-журнал ограничен 10 МБ × 3 файла; для workers, PostgreSQL и Redis — 5 МБ × 2 файла. Сборщик логов должен читать ротацию без предположения о бесконечном локальном файле.
