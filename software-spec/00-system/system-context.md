# Контекст системы

## Граница Clean Pay

Clean Pay — веб-кабинет, который связывает пользователя с учётной записью, подпиской и платёжными возможностями Remnashop/Remnawave. Внутри логической границы находятся функции доступа и идентичности, подписки, платежей и эксплуатации. PostgreSQL и Redis являются собственными хранилищами Clean Pay, но показаны отдельно, потому что их интерфейсы и эксплуатационный жизненный цикл должны быть воспроизведены.

```text
Пользователь и браузер
  │ страницы, формы, переходы, HTTP, cookie, WebAuthn, PWA-хранилища
  ▼
Reverse proxy / HTTPS-терминатор
  │ HTTP(S), forwarding-заголовки
  ▼
┌───────────────────────────────────────────────────────────────────┐
│                           CLEAN PAY                               │
│                                                                   │
│  Доступ и идентичность ─ Подписка ─ Платежи ─ Эксплуатация       │
│                  журналирование и аудит                           │
└───────────────────────────────────────────────────────────────────┘
  │          │           │          │          │          │
  │          │           │          │          │          └─► Mailpit API
  │          │           │          │          │              (только необязательная проверка готовности)
  │          │           │          │          └────────────► Turnstile
  │          │           │          └───────────────────────► Telegram OIDC
  │          │           └──────────────────────────────────► Remnawave API
  │          └──────────────────────────────────────────────► Remnashop public/admin API
  ├─────────────────────────────────────────────────────────► PostgreSQL Clean Pay
  └─────────────────────────────────────────────────────────► Redis Clean Pay

Remnashop ──► SMTP-провайдер ──► почтовый ящик пользователя
Remnashop ──► Telegram Bot API
Remnashop ──► Remnawave
Remnashop ──► платёжный провайдер
Пользовательский браузер ──► платёжный URL провайдера
Платёжный провайдер ──► Remnashop; затем пользователь возвращается в Clean Pay

Планировщик сверки ──► внутренний платёжный интерфейс Clean Pay
Процесс очистки ─────► PostgreSQL Clean Pay
Оператор ────────────► развёртывание, миграции, проверки готовности,
                       резервное копирование и восстановление
```

## Почему почта показана отдельно

Clean Pay не открывает SMTP-соединение сам. Он вызывает операции e-mail в Remnashop, а Remnashop формирует и доставляет письмо через настроенный SMTP-сервис. Поэтому это не один интерфейс, а цепочка из трёх самостоятельных границ:

1. Clean Pay → Remnashop: запросить код, подтвердить код, начать смену адреса.
2. Remnashop → SMTP: передать письмо для доставки.
3. SMTP-провайдер → пользователь: доставить письмо в почтовый ящик.

В среде разработки SMTP-провайдер заменён Mailpit. Clean Pay дополнительно может читать только признак доступности Mailpit через HTTP API; тестовый `smtp-log` получает webhook Mailpit и запрашивает тело сообщения через Mailpit API.

## Прямые и косвенные зависимости

| Участник | Связь с Clean Pay | Назначение |
|---|---|---|
| Браузер | прямая, двусторонняя | пользовательский интерфейс, cookie, WebAuthn, PWA, перенаправления |
| Reverse proxy | прямая входящая | HTTPS-терминация и доставка HTTP-запросов |
| PostgreSQL Clean Pay | прямая, двусторонняя | долговечное состояние продукта |
| Redis Clean Pay | прямая, двусторонняя | ограничение частоты и кэш эксплуатационной готовности |
| Remnashop public API | прямая, двусторонняя | аутентификация, профиль, тарифы, подписка, устройства и пользовательские платежи |
| Remnashop admin API | прямая, двусторонняя | объединение владельцев и фоновая сверка платежей |
| Remnawave API | прямая исходящая | проверка актуальной ссылки активной подписки |
| Telegram OIDC | прямая, двусторонняя через браузер и backend | вход и получение проверяемой Telegram-идентичности |
| Turnstile | прямая исходящая | проверка антибот-токена |
| Mailpit API | прямая исходящая, только при настройке | необязательная проверка готовности тестовой почты |
| SMTP-провайдер | косвенная через Remnashop | фактическая отправка e-mail |
| Telegram Bot API | косвенная через Remnashop | работа Telegram-бота и WebApp-контура Remnashop |
| Платёжный провайдер | косвенная через Remnashop и браузер | приём платежа и изменение его статуса |
| Remnawave со стороны Remnashop | косвенная | управление подпиской в авторитетной системе |
| Планировщик сверки | прямая входящая, внутренняя | повторная проверка неоднозначных платежей |
| Процесс очистки | внутренний эксплуатационный | удаление устаревших служебных данных |

Полный перечень транспортных операций и тестовых замен должен находиться в разделе внешних интеграций и в матрицах трассируемости. Укрупнение нескольких сервисов в одну стрелку недопустимо.
