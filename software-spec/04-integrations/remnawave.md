# Интеграция с Remnawave

## Назначение и общие правила

Clean Pay использует Remnawave только для чтения живого состояния подписки и подтверждённой ссылки подключения. Базовый адрес и Bearer token обязательны в production и должны настраиваться парой.

Каждый запрос: `GET`, `Accept: application/json`, `Authorization: Bearer <token>`, без кэша, таймаут 10 секунд, без автоматического повтора.

## RW-001: пользователь по UUID

```http
GET {BASE}/api/users/{url-encoded-user_remna_id}
```

Ожидаемый корень: `{response: RemnawaveUser|null}`.

## RW-002: пользователи по e-mail

```http
GET {BASE}/api/users/by-email/{url-encoded-email}
```

Ожидаемый корень: `{response: RemnawaveUser[]|null}`; `null` трактуется как пустой массив.

## RW-003: пользователи по Telegram ID

```http
GET {BASE}/api/users/by-telegram-id/{url-encoded-id}
```

Ответ аналогичен RW-002.

## Используемые поля `RemnawaveUser`

| Поле | Тип | Nullable | Правило |
|---|---|---:|---|
| `uuid` | string | да | внешние пробелы удаляются; обязан совпасть с ожидаемым владельцем |
| `status` | string | да | живым считается только точное `ACTIVE` |
| `email` | string | да | внешние пробелы удаляются, регистр понижается при сопоставлении |
| `telegramId` | string/integer | да | сравнение после строковой нормализации |
| `expireAt` | date-time string | да | отсутствие допускается; при наличии дата обязана быть валидной и будущей |
| `subscriptionUrl` | string | да | предпочтительное имя |
| `subscription_url` | string | да | совместимый alias |

Ссылка принимается только как непустой абсолютный HTTP(S) URL.

## Алгоритм разрешения ссылки

1. При наличии `user_remna_id` сначала выполняется RW-001.
2. Результат принимается только при точном UUID, `ACTIVE`, неистёкшем `expireAt` и валидной ссылке.
3. Если первичный поиск не дал ссылки, выполняются доступные RW-002/RW-003.
4. Результаты группируются по непустому UUID; остаются только живые пользователи с ожидаемыми e-mail/Telegram-идентичностями.
5. Ссылка возвращается только если существует ровно один подходящий UUID и ровно одна уникальная ссылка. Любая неоднозначность даёт `null`.

`404`, иной не-2xx, невалидный JSON, сетевая ошибка, таймаут и отсутствие настройки деградируют в `null`, а не пробрасываются как техническая ошибка. При формировании текущей подписки Clean Pay преобразует отсутствие проверенной живой ссылки в продуктовую ошибку `SUBSCRIPTION_URL_UNAVAILABLE`; значение `url` из Remnashop не используется как резервное.

## RW-004: metadata готовности

Операция выполняется только если отдельно задан URL проверки готовности:

```http
GET {READINESS_BASE}/api/system/metadata
Accept: application/json
Authorization: Bearer <token>
```

Тело не проверяется; любой 2xx — `ok`. Таймаут отдельной проверки 5 секунд, общий предел 8 секунд. Если URL отсутствует, проверка не включается.

## Тестовая замена

`remnawave-mock` отвечает фиксированными metadata на `/api/system/metadata`. На прочие пути он возвращает `{response:null}`. Поэтому dev-имитатор подтверждает доступность и безопасную ветку отсутствия живой подписки, но не эмулирует полный набор операций Remnawave.
