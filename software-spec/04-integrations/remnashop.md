# Интеграция с Remnashop

## Назначение

Remnashop — авторитетная внешняя система для пользовательских учётных данных, профиля, тарифных предложений, состояния подписки, устройств, промокодов, создания платежей и истории платежей. Admin API дополнительно используется для объединения владельцев и фоновой сверки неоднозначных платежей.

## Транспортные границы

| Граница | Базовый адрес | Авторизация |
|---|---|---|
| Public API | настраиваемый адрес, обязательно оканчивается `/api/v1/public` | auth-cookie Remnashop для пользовательских операций; без cookie для регистрации и входа |
| Admin API | отдельный настраиваемый адрес либо тот же origin с суффиксом `/api/v1/admin` | `x-api-key: <REMNASHOP_API_KEY>` |

Все запросы используют HTTP(S), `Accept: application/json`, `Cache-Control` на стороне клиента эквивалентен режиму без кэширования. При наличии JSON-тела отправляется `Content-Type: application/json`. Общий таймаут — 15 секунд; операции recovery — 10 секунд. Автоматического повтора общего запроса нет.

## Пользовательская авторизация Remnashop

Clean Pay не передаёт собственную session-cookie во внешний сервис. Он хранит отдельную пару токенов Remnashop и формирует исходящий заголовок:

```http
Cookie: access_token=<upstream access JWT>; refresh_token=<upstream refresh JWT>
```

Обычно достаточно `access_token`; для обновления отправляется только `refresh_token`. Ответы регистрации, входа, Telegram-аутентификации, обновления токенов и смены пароля обязаны содержать обе cookie `access_token` и `refresh_token` в `Set-Cookie`. Отсутствие любой из них является нарушением upstream-контракта и трактуется как ошибка шлюза.

## Идемпотентность платежей

Для создания покупки/продления и для recovery Clean Pay отправляет `Idempotency-Key`. Значение, переданное в Remnashop, является внутренним стабильным ключом конкретной локальной платёжной операции; повтор той же операции обязан использовать тот же ключ. До создания платежа Clean Pay повторно получает актуальные offers и сверяет подтверждённые пользователем цену, валюту и версию предложения. Поля этой локальной сверки не пересылаются в Remnashop; наружу уходят только выбранный тариф/срок/шлюз и серверный `return_url`.

## Ошибки и деградация

- Любой неуспешный HTTP-ответ разбирается как JSON с полем `detail`; если JSON отсутствует, используется текст тела.
- Успешный ответ с невалидным JSON является ошибкой внешнего контракта.
- Сетевая ошибка и таймаут означают недоступность Remnashop.
- `401` входа отличается от общей потери авторизации; `401` смены пароля означает неверный текущий пароль.
- Ошибки email-кода, промокода, подписки, устройств, payment idempotency и unavailable gateway нормализуются в стабильные ошибки Clean Pay.
- Неопределённый результат после пересечения платёжной границы никогда не превращается в обычный отказ: операция сохраняется и переводится в сверку либо ручную проверку.

## Допустимые повторы

| Сценарий | Повтор |
|---|---|
| Обычный public/admin запрос | нет |
| Отправка кода подтверждения e-mail | до 3 попыток только для распознанной временной ошибки доставки; задержки 300 и 600 мс |
| Обновление upstream access-token | одна координированная попытка на истёкшую сессию |
| Смена пароля при рассинхронизации токена | после обновления токена допускается одна повторная попытка |
| Неоднозначный платёж | повтор не создаёт новый платёж; выполняется recovery с тем же ключом |

## Важные отсутствующие вызовы

- Clean Pay не вызывает Remnashop `POST /auth/logout`; локальный logout отзывает собственную Clean Pay session и удаляет локально сохранённый upstream token bundle.
- Clean Pay не вызывает password reset endpoints Remnashop; текущий продукт предоставляет login/register/change-password, но не отдельный forgot-password flow.
- Clean Pay не вызывает trial endpoints Remnashop.
- Clean Pay не принимает Remnashop webhook и не публикует ему webhook.
- Clean Pay не управляет настройками payment gateways Remnashop.

Эти отсутствия являются частью текущей системной границы: их нельзя объявлять существующими функциями новой реализации без отдельного решения.

## Связанные внешние системы Remnashop

Функции Clean Pay косвенно зависят от следующих границ Remnashop:

- SMTP-провайдер — доставка кода подтверждения;
- Telegram Bot API — работа Telegram-бота и подпись Telegram Login payload;
- Remnawave — фактическое управление подписками и устройствами;
- один из активных платёжных шлюзов — выдача `payment_url` и последующее изменение статуса платежа;
- собственные PostgreSQL и Valkey, worker и scheduler Remnashop.

Эти участники не должны ошибочно считаться внутренними частями Clean Pay. Точные 30 исходящих HTTP-операций Clean Pay → Remnashop и полная нормализация ошибок/служебных заголовков зафиксированы отдельно.
