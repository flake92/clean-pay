# PostgreSQL и Redis как внешние технические интерфейсы

## PostgreSQL Clean Pay

PostgreSQL является авторитетным долговечным хранилищем собственного состояния Clean Pay. Production использует отдельную БД, пользователя и именованный том; порт наружу не публикуется. Проверка готовности выполняет `SELECT 1` с отдельным соединением и timeout 5 секунд.

Транзакции обязаны сохранять атомарность сессий, одноразовых состояний, привязки идентичностей, payment operation, истории, аудита и удаления. Для конфликтующих изменений владельца и платежей используются блокировки строк и детерминированный порядок захвата. Физический контракт полей, ключей, индексов и ограничений описывается в разделе данных.

Миграции production применяются только последовательной командой deploy до запуска новой версии. Процесс очистки подключается к той же БД и удаляет только истёкшие служебные записи; платёжные записи не удаляются его текущей политикой.

## Redis Clean Pay

Redis доступен по `REDIS_URL`, production-образ использует AOF. Клиент Clean Pay реализует RESP поверх TCP. Существенные команды включают readiness `PING`, операции счётчиков/TTL для rate limiting и общую запись кэша публичной готовности.

### Транспорт и ограничения

- `redis://` использует TCP, `rediss://` — TLS с SNI по hostname;
- default port 6379 для plain и 6380 для TLS;
- connect timeout 2 секунды, единый deadline команды вместе с AUTH/SELECT — 3 секунды;
- максимальный ответ 1 MiB;
- поддерживаемые RESP-ответы: simple string, error, integer, bulk string/null и массив;
- каждое логическое обращение открывает отдельное соединение и закрывает его после ответа;
- при username/password выполняется `AUTH username password`, только password — `AUTH password`;
- непустой path URL выбирает БД командой `SELECT {db}`.

### Фактически используемые команды

| ID | Команда | Назначение | Ожидаемый ответ |
|---|---|---|---|
| REDIS-001 | `PING` | readiness | точная строка `PONG` |
| REDIS-002 | `EVAL <script> 1 <key> <windowSeconds>` | атомарно `INCR`; при первом значении `EXPIRE` | integer counter |
| REDIS-003 | `TTL <key>` | вычислить `Retry-After` | положительное integer либо fallback к размеру окна |
| REDIS-004 | `SET clean-pay:health:readiness:v1 <json> EX 120` | общий кэш последней подробной готовности | успешный Redis-ответ; ошибка деградирует readiness |
| REDIS-005 | `GET clean-pay:health:readiness:v1` | публичная готовность без дорогого fan-out | JSON string либо null |

Lua script REDIS-002 имеет точную семантику: увеличить ключ; если новое значение равно 1, назначить TTL `windowSeconds`; вернуть count. Ключ содержит нормализованное имя действия и HMAC-SHA256 дайджесты e-mail, Telegram ID и IP, поэтому исходные идентификаторы в Redis не раскрываются.

Rate-limit в большинстве защищённых операций fail-closed при недоступности Redis. Операция identify является явным исключением и может продолжить работу после ошибки инфраструктуры. Публичный readiness сначала использует общий Redis cache, затем локальный process cache; устаревший результат старше 90 секунд не считается допустимым. Запись общей готовности имеет TTL 120 секунд.

## Раздельность хранилищ

Dev-среда содержит также `remnashop-postgres` и `remnashop-cache` (Valkey). Они принадлежат Remnashop и не могут быть объединены с PostgreSQL/Redis Clean Pay. Их данные, тома и lifecycle независимы.

## Тестовые контуры

Интеграционные проверки используют реальные PostgreSQL/Redis-соединения там, где проверяются locking, deletion и timeout semantics. Mock не должен заменять транзакционное владение или атомарность production-хранилищ.
