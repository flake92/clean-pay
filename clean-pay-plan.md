# Техническое задание для LLM-разработчика  
## Web-кабинет CleanVPN на Next.js + BFF + Prisma + Nginx

## 1. Цель разработки

Необходимо разработать отдельный web-кабинет CleanVPN для клиентов, которые не используют Telegram-бота.

Web-кабинет должен позволять пользователю:

- зарегистрироваться по e-mail;
- подтвердить e-mail кодом;
- войти в личный кабинет;
- посмотреть доступные тарифы;
- купить VPN-подписку;
- продлить действующую подписку;
- посмотреть текущую подписку;
- получить ссылку или данные подключения VPN;
- управлять подключёнными устройствами, если это доступно через API Remnashop;
- обратиться в поддержку.

Сайт должен работать на домене:

`oplata.clear-vpn.org`

---

## 2. Общая архитектура

Проект должен быть реализован как отдельный web-сервис, не встроенный в Remnashop.

Архитектура:

1. **Nginx reverse proxy**
   - принимает внешний HTTPS-трафик;
   - проксирует запросы в Next.js-приложение;
   - обслуживает домен `oplata.clear-vpn.org`;
   - работает с SSL-сертификатом.

2. **Next.js-приложение**
   - содержит frontend;
   - содержит backend/BFF-слой;
   - отвечает за пользовательский интерфейс;
   - принимает запросы от браузера;
   - обращается к Remnashop API только с серверной стороны.

3. **BFF внутри Next.js**
   - является промежуточным backend-слоем между frontend и Remnashop;
   - проксирует регистрацию, вход, профиль, тарифы, подписки, оплаты и устройства;
   - скрывает внутренний адрес Remnashop API;
   - управляет web-сессией пользователя;
   - использует собственные JWT/cookie для авторизации в web-кабинете.

4. **Remnashop**
   - остаётся основной системой бизнес-логики;
   - управляет пользователями, тарифами, подписками, оплатами, webhook-ами и VPN-доступами;
   - все операции выполняются через публичный API Remnashop.

5. **Собственная cache DB**
   - отдельная PostgreSQL-база web-кабинета;
   - используется только для сопутствующей информации;
   - не заменяет БД Remnashop;
   - не хранит основную бизнес-логику подписок и оплат.

6. **Prisma ORM**
   - используется только для собственной cache DB web-кабинета;
   - не подключается к базе Remnashop.

7. **PrimeReact Sakai**
   - shadcn/ui отменён;
   - интерфейс собирается на PrimeReact и физически перенесённых Sakai layout/styles/assets;
   - нельзя вводить самодельную UI-систему вместо Sakai/PrimeReact.

---

## 3. Главные технические ограничения

- Нельзя менять структуру БД Remnashop.
- Нельзя добавлять миграции в БД Remnashop.
- Нельзя напрямую писать в PostgreSQL Remnashop.
- Нельзя обращаться к Remnashop из frontend напрямую.
- Все бизнес-операции должны идти через API Remnashop.
- Собственная БД web-кабинета используется только для вспомогательной информации.
- Web-кабинет не должен самостоятельно подтверждать оплаты.
- Web-кабинет не должен обходить бизнес-логику Remnashop.
- Секреты не должны попадать во frontend-код.
- Секреты не должны попадать в Git.
- JWT web-кабинета должны быть собственными.
- Аутентификация пользователя должна проксироваться через Remnashop.
- Все чувствительные auth-действия должны быть защищены Cloudflare Turnstile: регистрация, вход, запрос e-mail-кода, подтверждение e-mail-кода, Telegram login/link и привязка `tg_id`.

---

## 4. Основные API Remnashop, которые нужно использовать

Базовый путь API:

`/api/v1/public`

### Авторизация

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`
- `GET /auth/me`
- `POST /auth/change-password`
- `POST /auth/email/change`
- `POST /auth/email/request-verification`
- `POST /auth/email/confirm`

### Тарифы

- `GET /plans/public`

### Подписка

- `GET /subscription/current`
- `GET /subscription/offers`
- `POST /subscription/purchase`
- `POST /subscription/extend`

### Устройства

- `GET /subscription/devices`
- `DELETE /subscription/devices/{hwid}`

Оплата должна создаваться через Remnashop. Web-кабинет получает ссылку оплаты и перенаправляет пользователя на платёжную страницу. Подтверждение оплаты и webhook-и обрабатывает Remnashop.

## Шаг 5. Реализовать схему авторизации web-кабинета

Необходимо реализовать авторизацию через Remnashop с собственной web-сессией.

Схема должна быть такой:

1. Пользователь вводит e-mail и пароль.
2. Frontend получает Cloudflare Turnstile token.
3. BFF проверяет Turnstile token на сервере.
4. BFF отправляет данные в Remnashop.
5. Remnashop подтверждает пользователя.
6. Web-кабинет создаёт собственную сессию.
7. Web-кабинет выдаёт пользователю собственные JWT/cookie.
8. Все последующие запросы идут через BFF.
9. BFF проверяет web-сессию и при необходимости обращается к Remnashop.

Необходимо предусмотреть:

- регистрацию;
- вход;
- выход;
- обновление сессии;
- получение текущего пользователя;
- защищённые страницы;
- истечение сессии;
- безопасное хранение refresh token;
- HttpOnly cookie;
- Secure cookie;
- SameSite-политику.
- Cloudflare Turnstile на регистрации и входе.
- Cloudflare Turnstile на Telegram login/link flow перед стартом OIDC.

Результат шага: пользователь может авторизоваться в web-кабинете через Remnashop, но сама web-сессия управляется новым проектом.

Необходимо предусмотреть:

- повторную отправку кода;
- ограничение частоты отправки;
- обработку неверного кода;
- обработку истёкшего кода;
- нейтральные сообщения, не раскрывающие наличие пользователя в базе;
- блокировку покупки до подтверждения e-mail.
- Cloudflare Turnstile на запросе кода и подтверждении кода.

Результат шага: e-mail подтверждается через Remnashop, а web-кабинет корректно управляет пользовательским сценарием.

## Шаг 8. Реализовать покупку подписки

Необходимо реализовать процесс покупки подписки.

Процесс:

1. Пользователь выбирает тариф.
2. Пользователь выбирает длительность.
3. Пользователь переходит на страницу оплаты.
4. Web-кабинет проверяет, что e-mail подтверждён.
5. BFF отправляет запрос покупки в Remnashop.
6. Remnashop создаёт платёж.
7. Web-кабинет получает ссылку оплаты.
8. Пользователь перенаправляется на страницу платёжной системы.
9. После возврата пользователь попадает на страницу результата оплаты.

Важно:

- web-кабинет не подтверждает оплату;
- web-кабинет не обрабатывает webhook-и;
- источник истины по оплате — Remnashop;
- источник истины по подписке — Remnashop.

Результат шага: пользователь может купить подписку через процесс, управляемый Remnashop.

Базовый интерфейс на Next.js и PrimeReact Sakai уже реализован раньше исходной очереди и считается актуальной базой проекта. На шаге 17 нужно провести финальную полировку, выравнивание UX и проверку полноты страниц.

Обязательные страницы:

- `/` — главная;
- `/register` — регистрация;
- `/login` — вход;
- `/verify-email` — подтверждение e-mail;
- `/tariffs` — тарифы;
- `/payment` — оплата;
- `/payment/success` — успешная оплата;
- `/payment/fail` — неуспешная оплата;
- `/payment/pending` — оплата в обработке;
- `/cabinet` — личный кабинет;
- `/devices` — устройства;
- `/profile` — профиль;
- `/support` — поддержка.

Интерфейс должен быть:

- адаптивным;
- удобным на телефоне;
- визуально связанным с CleanVPN;
- визуально единым с PrimeReact Sakai;
- понятным для пользователя без технических знаний;
- с минимальным количеством шагов до оплаты.

Результат шага: пользовательский интерфейс web-кабинета финально проверен, отполирован и согласован с PrimeReact Sakai-базой.

## Шаг 18. Подготовить Docker-сборку

Необходимо подготовить контейнеризацию проекта.

Должны быть контейнеры:

- Next.js web app;
- PostgreSQL cache DB;
- Nginx reverse proxy.

Также должен быть отдельный compose-файл для web-кабинета.

Требования:

- compose не должен заменять compose Remnashop;
- web-кабинет должен разворачиваться отдельно;
- Remnashop должен быть доступен web-кабинету по внутреннему адресу или через reverse proxy;
- секреты должны передаваться через env;
- данные PostgreSQL должны храниться в volume.

Результат шага: проект можно развернуть как отдельный набор контейнеров.

## Шаг 19. Настроить Nginx reverse proxy

Необходимо подготовить конфигурацию Nginx.

Nginx должен:

- обслуживать домен `oplata.clear-vpn.org`;
- принимать HTTPS;
- проксировать запросы в Next.js;
- корректно передавать заголовки;
- поддерживать healthcheck;
- не раскрывать внутренние адреса сервисов;
- использовать SSL-сертификат.

Результат шага: сайт доступен через Nginx по нужному домену.

## Шаг 20. Подготовить SSL через Cloudflare

Необходимо подготовить выпуск SSL-сертификата через Cloudflare API.

Нужно предусмотреть:

- создание или проверку DNS-записи;
- выпуск Origin CA сертификата;
- сохранение сертификата на сервере;
- сохранение private key только на сервере;
- подключение сертификата к Nginx;
- перезагрузку Nginx после установки сертификата.

Результат шага: `oplata.clear-vpn.org` работает по HTTPS через Cloudflare Full Strict.

# 6. MVP считается готовым, если

- сайт открывается на `oplata.clear-vpn.org`;
- пользователь может зарегистрироваться;
- пользователь может подтвердить e-mail;
- пользователь может войти;
- пользователь видит тарифы;
- пользователь может выбрать тариф;
- пользователь может перейти к оплате;
- после оплаты пользователь видит подписку;
- пользователь видит VPN-ссылку или данные подключения;
- пользователь может продлить подписку;
- пользователь может управлять устройствами, если API это поддерживает;
- frontend не обращается к Remnashop напрямую;
- все запросы к Remnashop идут через BFF;
- используется собственная web-сессия на JWT/cookie;
- собственная БД используется только для сопутствующих данных;
- БД Remnashop не изменяется;
- секреты не попадают в frontend и Git;
- проект разворачивается отдельными контейнерами;
- есть документация по запуску и откату.
