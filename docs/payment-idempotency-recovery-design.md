# Восстановление payment idempotency

## Решение

Серверная дедупликация разных client idempotency keys по одному payload не вводится. Для Clean Pay легитимны повторные покупка и продление, а одинаковый payload не доказывает, что это повтор одного платежного намерения. Особенно опасны смена цены, `offer_version`, повторная покупка после успешного платежа и `OUTCOME_UNKNOWN` у провайдера.

Вместо эвристического окна реализовано безопасное восстановление уже существующей незавершённой операции:

- `GET /api/bff/payments/status` без `payment_id` и `operation_id` ищет последнюю операцию текущего пользователя в `DISPATCHING` или `OUTCOME_UNKNOWN`;
- поиск строго ограничен `userId` текущей сессии;
- ответ использует существующую state machine и не создаёт новую `PaymentOperation`;
- callback/status UI уже вызывает endpoint без идентификаторов, если browser storage отсутствует, поэтому операция восстанавливается и на другом устройстве;
- операции `SUCCEEDED`, `FAILED_FINAL` и `READY` не считаются активными и не блокируют легитимную новую покупку.

## Подтверждённое поведение

Unit-тест `payment-idempotency.test.ts` фиксирует текущий контракт: два разных UUID idempotency key с одинаковым fingerprint создают две операции. Это сознательно не меняется без подтверждённого upstream-контракта.

Route-тест `bff-routes.test.ts` подтверждает восстановление последней активной операции без browser-local identifiers.

## Fault-injection matrix

Перед возможным введением cross-key deduplication нужен отдельный disposable Remnashop/payment-provider стенд. На production-подобном стенде с реальным платёжным провайдером эти проверки выполнять нельзя.

| Сбой | Что нужно подтвердить upstream | Текущее безопасное поведение Clean Pay |
|---|---|---|
| TCP reset до получения HTTP response | Был ли запрос принят | `OUTCOME_UNKNOWN`, reconciliation, новая операция не предлагается status UI |
| Timeout после отправки body | Может ли провайдер создать платёж | Та же операция остаётся восстанавливаемой |
| HTTP 5xx после фактического создания | Стабильность upstream idempotency key | Повтор выполняется только с сохранённым `upstreamKey` |
| Потеря browser storage | Можно ли безопасно сопоставить intent без client key | Активная операция находится по сессии; автоматического cross-key merge нет |
| Изменение цены/offer version | Можно ли считать intent эквивалентным | Считается новым intent; серверная offer validation остаётся обязательной |

## Критерий для будущего cross-key guard

Он допустим только если Remnashop документирует невозможность повторного списания или предоставляет устойчивый provider-side operation lookup. До этого fingerprint нельзя использовать как финансовый уникальный ключ.
