Изменение аутентификационной обвязки

Переделываем аутентификационную обвязку с учётом WebAuthn и требований Remnashop.

Remnashop, к которому мы проксируем пользовательские действия, требует, чтобы пользователь был идентифицирован одним из следующих способов:

* через подтверждённый email;
* через Telegram ID.

Поэтому предлагается следующая логика.

Страница входа

На странице входа пользователь должен выбрать один из вариантов:

* войти по email;
* войти через Telegram.

Вход по email

Если пользователь выбирает вход по email, сначала он вводит свой email.

После этого отправляется код подтверждения на указанный адрес.

На этом этапе не нужно выполнять полноценную инициализацию пользователя и считать email подтверждённым способом входа.

Сначала пользователь должен ввести код подтверждения.

Только после успешного подтверждения кода email считается подтверждённым, и можно продолжать дальнейший сценарий авторизации и инициализации пользователя согласно общей логике WebAuthn и интеграции с Remnashop.

Сценарий входа по email:

1. Пользователь вводит email.
2. Система отправляет код подтверждения.
3. Пользователь вводит код.
4. После успешной проверки кода email считается подтверждённым.
5. Далее продолжается основной сценарий авторизации.

Вход через Telegram

Пользователь также может войти через Telegram.

В этом случае пользователь идентифицируется через Telegram ID, так как Remnashop поддерживает такой способ авторизации.

Если пользователь вошёл через Telegram, но у него не привязан email, в интерфейсе должна отображаться постоянная информационная плашка.

Текст плашки:

Рекомендуем привязать email, чтобы не потерять доступ к аккаунту.

Плашка должна содержать ссылку на раздел связанных учётных записей, где пользователь сможет привязать email.

Связанные учётные записи

Вместо отдельного действия или раздела Подтвердить email вводим раздел:

Связанные учётные записи

В этом разделе пользователь должен видеть все способы входа, связанные с его аккаунтом:

* email;
* Telegram.

Для каждого способа входа необходимо отображать его текущее состояние:

* привязан;
* не привязан;
* ожидает подтверждения, если это применимо.

В этом разделе пользователь может:

* привязать email;
* подтвердить email кодом;
* привязать Telegram;
* удалить привязанный email;
* удалить привязанный Telegram.

Ограничение на удаление способов входа

Пользователь не должен иметь возможность удалить последний оставшийся способ входа.

То есть в системе всегда должен оставаться хотя бы один подтверждённый способ доступа к аккаунту.

Примеры:

* если у пользователя привязан только email, удалить email нельзя;
* если у пользователя привязан только Telegram, удалить Telegram нельзя;
* если у пользователя привязаны и email, и Telegram, можно удалить один из способов входа, но второй должен остаться.

Это ограничение необходимо, чтобы пользователь не потерял доступ к аккаунту полностью.

Итоговая логика

У пользователя всегда должен быть хотя бы один подтверждённый способ входа:

* подтверждённый email;
* или привязанный Telegram ID.

Email считается полноценным способом входа только после подтверждения кодом.

Telegram может использоваться как самостоятельный способ входа, но если у пользователя не привязан email, интерфейс должен рекомендовать привязать его через постоянную информационную плашку.

Раздел Подтвердить email заменяется на раздел Связанные учётные записи, где пользователь управляет привязкой email и Telegram.
