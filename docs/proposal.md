Изменение аутентификационной обвязки

Переделываем аутентификационную обвязку с учётом обязательного WebAuthn/passkey и требований Remnashop.

Remnashop, к которому мы проксируем пользовательские действия, требует, чтобы пользователь был идентифицирован одним из следующих способов:

* через подтверждённый email;
* через Telegram ID.

При этом в Clean Pay полноценная авторизация выполняется только через WebAuthn/passkey.

Email и Telegram используются для первичной идентификации, привязки Remnashop identity и восстановления доступа, но не считаются заменой passkey.

Проект находится в тестовой среде, поэтому допускается менять текущую auth-модель, схему данных и пользовательские сценарии без поддержки старого поведения как финального.

Целевая модель

Clean Pay разделяет три сущности:

* identity: подтверждённый email или Telegram ID, необходимые для Remnashop;
* authenticator: WebAuthn/passkey, обязательный для входа в Clean Pay;
* session: полноценная Clean Pay session, которая создаётся только после успешного WebAuthn.

Полноценная Clean Pay session создаётся только после WebAuthn.

До успешного WebAuthn пользователь не должен получать доступ ни к кабинету, ни к Remnashop-действиям, ни к чувствительным данным.

Страница входа

Основной способ входа:

* войти с ключом доступа.

Дополнительные сценарии:

* восстановить доступ через Telegram;
* восстановить доступ через email.

Telegram и email не должны отображаться как равноправные основные способы входа. Они используются для восстановления доступа, первичной привязки или добавления passkey на новом устройстве.

Вход с passkey

Обычный сценарий:

1. Пользователь нажимает Войти с ключом доступа.
2. Браузер запускает WebAuthn challenge.
3. Пользователь подтверждает вход через Face ID, Touch ID, Windows Hello, PIN или другой механизм устройства.
4. Clean Pay проверяет WebAuthn assertion.
5. Создаётся full Clean Pay session.
6. Пользователь получает доступ к кабинету и Remnashop-действиям.

Вход через Telegram

Telegram используется как дополнительный способ идентификации и восстановления доступа.

После успешного Telegram OIDC пользователь не получает полноценную Clean Pay session.

После Telegram необходимо сразу создать или подтвердить passkey.

Сценарий:

1. Пользователь выбирает восстановление или вход через Telegram.
2. Telegram OIDC подтверждает Telegram ID.
3. Clean Pay находит или подготавливает пользователя по Telegram ID.
4. Пользователь должен создать passkey на текущем устройстве.
5. Только после успешного создания passkey создаётся full Clean Pay session.
6. Пользователь попадает в кабинет.

Если пользователь не создал passkey, полноценный пользователь и полноценная session не должны оставаться в базе как завершённая регистрация.

Для этого сценарий должен быть транзакционным:

* временные состояния хранятся отдельно от завершённых пользователей или имеют TTL;
* Remnashop identity не считается окончательно привязанной до успешного создания passkey;
* при отмене, истечении TTL или ошибке WebAuthn временные записи очищаются;
* база не должна захламляться незавершёнными Telegram-попытками.

Вход и восстановление через email

Email используется как recovery-сценарий, в том числе для возврата доступа при утере passkey.

Сценарий:

1. Пользователь выбирает восстановление через email.
2. Пользователь вводит email.
3. Clean Pay инициирует отправку кода подтверждения через Remnashop.
4. Пользователь вводит код.
5. После успешной проверки кода email считается подтверждённым recovery identity.
6. Пользователь должен создать passkey на текущем устройстве.
7. Только после успешного создания passkey создаётся full Clean Pay session.

Email-код не создаёт полноценную Clean Pay session сам по себе.

Если email-доставка недоступна или код не приходит, интерфейс должен показывать плашку с обращением в техническую поддержку.

Плашка технической поддержки

Плашка нужна для случаев, когда пользователь не может восстановить доступ через email или Telegram.

Текст должен быть прикладным:

Если код не приходит или вы потеряли доступ к Telegram, обратитесь в техническую поддержку. Мы поможем восстановить доступ к аккаунту.

Плашка должна содержать доступный способ связи с поддержкой.

Связанные учётные записи

Раздел Связанные учётные записи должен показывать:

* ключи доступа;
* email;
* Telegram.

Для каждого элемента нужно отображать состояние:

* привязан;
* не привязан;
* ожидает подтверждения, если применимо;
* последний активный, если удаление запрещено.

В этом разделе пользователь может:

* добавить новый passkey;
* удалить passkey, если он не последний;
* привязать email;
* подтвердить email кодом;
* привязать Telegram;
* удалить привязанный email;
* удалить привязанный Telegram.

Passkey

У одного пользователя может быть несколько passkey.

Примеры:

* телефон;
* ноутбук;
* рабочий компьютер;
* аппаратный security key.

Последний passkey удалить нельзя.

У аккаунта всегда должен быть хотя бы один активный passkey.

Telegram и email не считаются заменой passkey. Они нужны только для восстановления доступа или привязки нового устройства.

Если пользователь хочет удалить passkey со старого устройства, он должен сначала добавить новый passkey на другом устройстве.

Ограничение на удаление способов доступа

Пользователь не должен иметь возможность удалить последний активный passkey.

Email и Telegram можно удалять только если это не ломает требования Remnashop и не оставляет пользователя без recovery-сценария, принятого продуктовой политикой.

Приоритетное правило:

* passkey обязателен всегда;
* email или Telegram нужны для Remnashop identity и recovery;
* email/Telegram не заменяют passkey.

Remnashop refresh token

Если пользователь успешно прошёл WebAuthn, но сохранённый Remnashop refresh token истёк или стал невалидным, Clean Pay не должен молча продолжать Remnashop-действия.

В этом случае пользователь должен заново подтвердить Telegram или email.

После повторного подтверждения Clean Pay заново связывает пользователя с Remnashop identity и обновляет Remnashop tokens.

Telegram WebView

Passkey должен поддерживаться прямо внутри Telegram WebView.

Если конкретная платформа или версия WebView не поддерживает WebAuthn корректно, это нужно обработать отдельным UX-сценарием, но целевое требование остаётся: работа внутри Telegram WebView.

Итоговая логика

Полноценная Clean Pay session создаётся только после WebAuthn.

Remnashop identity обеспечивается через подтверждённый email или Telegram ID.

Telegram и email используются для:

* первичной идентификации;
* восстановления доступа;
* привязки Remnashop identity;
* добавления passkey на новом устройстве.

Telegram и email не являются полноценными способами входа без passkey.

После Telegram или email пользователь обязан создать или подтвердить passkey.

Без успешного WebAuthn пользователь не получает доступ ни к кабинету, ни к Remnashop-действиям.
